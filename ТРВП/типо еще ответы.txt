1. Прямые и обратные прокси-серверы: определение, назначение, применение, реализации
1. Прямой прокси-сервер (Forward Proxy)
Определение и назначение: Прямой прокси-сервер — это промежуточный узел, который действует от имени клиента. Он принимает запросы от пользователей внутри частной сети и пересылает их в общедоступную сеть (Internet).
Главное назначение: Сокрытие реального IP-адреса клиента от целевого сервера.
Применение: Используется для обеспечения анонимности, обхода географических ограничений или контроля доступа сотрудников к внешним ресурсам.
Реализация: В качестве прямого прокси может выступать сервер NGINX, который поддерживает работу на различных операционных системах, включая Linux, Windows, macOS и облачные платформы (AWS, Azure).
2. Обратный прокси-сервер (Reverse Proxy)
Определение и назначение: Обратный прокси-сервер — это компонент веб-системы, который располагается перед группой внутренних серверов и действует от их имени. В отличие от прямого прокси, он защищает и оптимизирует работу серверной части.
Главное назначение: Сокрытие реальных адресов внутренних серверов и балансировка нагрузки.
Применение в веб-системах: Обратные прокси являются основой архитектурного паттерна API Gateway (шлюз API), выполняя следующие задачи:
Маршрутизация (Routing): Перенаправление клиентских запросов на нужные внутренние сервисы.
Балансировка нагрузки (Load Balancing): Распределение трафика между несколькими экземплярами серверов для повышения отказоустойчивости и производительности.
Обслуживание статических запросов: Эффективная отдача файлов (HTML, CSS, изображения, React-bundle), что освобождает основные серверы приложений от этой нагрузки. То есть вместо большого количества запросов у нас статика отдается условно мгновенно, а остальное идет через апи.
Безопасность и аутентификация: Централизованная проверка JWT-токенов или OAuth2-авторизация до передачи запроса серверам.
Терминация SSL/TLS: Выполнение ресурсоемких операций шифрования и дешифрования трафика (HTTPS) на стороне прокси.

Основные реализации:
NGINX:
Высокопроизводительный сервер, использующий процессную архитектуру (один Master-процесс и несколько Worker-процессов, число которых обычно равно количеству ядер процессора).
Работает на инфраструктурном уровне (Edge Layer) и часто развертывается в Docker-контейнерах.
Поддерживает протоколы FastCGI/SCGI для связи с кодом на C#, Go, Python.
Ocelot:
Легкий .NET-фреймворк для создания шлюза на уровне приложения.
Поддерживает сложные алгоритмы балансировки, такие как Least Connection (наименьшее количество соединений) или Round Robin.
Включает механизм Circuit Breaker (прерыватель цепи) для временной остановки запросов к неработающим сервисам.
Envoy:
Используется в платформе Consul как «Sidecar-прокси» (сателлит сервиса) для взаимной TLS-аутентификации (mTLS) и умной маршрутизации.
Другие решения:
HAProxy и Apache HTTP Server часто применяются как специализированные балансировщики и прокси.
Облачные балансировщики (Azure Load Balancer, AWS Elastic Load Balancer).
Пример применения: В распределенной системе все запросы приходят на единую точку входа — шлюз Ocelot. Шлюз проверяет подлинность пользователя через JWT, после чего проксирует запрос к одному из внутренних микросервисов, чьи адреса скрыты от внешнего мира и динамически регистрируются в системе Consul.


2. Сервер NGINX: назначение, применение, серверные процессы, конфигурация, модули, журналы.
2. Сервер NGINX
NGINX — это высокопроизводительный HTTP-сервер и прокси-сервер с открытым исходным кодом, созданный Игорем Сысоевым в 2004 году. В современной веб-разработке он является ключевым компонентом веб-систем — распределенных совокупностей серверов, взаимодействующих для решения общих задач.
Назначение и применение
Forward proxy (прямой прокси): Используется на стороне клиента для сокрытия его реального IP-адреса.
Reverse proxy (обратный прокси): Применяется для сокрытия реальных адресов внутренних серверов и обеспечения безопасности.
Балансировщик нагрузки (Load Balancer): Распределяет входящий трафик между несколькими экземплярами серверов для повышения отказоустойчивости и производительности.
Обслуживание статических запросов: Крайне эффективен для отдачи статических файлов (HTML, CSS, изображения, React-bundle), что освобождает основные серверы приложений от этой нагрузки.
API Gateway (Шлюз API): Реализация паттерна шлюза на инфраструктурном уровне (Edge Layer — крайний уровень), обеспечивающая централизованную точку входа и управление трафиком.
Серверные процессы
Работа NGINX основана на разделении функций между специализированными процессами:
Master-процесс (один): Отвечает за чтение конфигурационных файлов и управление рабочими процессами.
Worker-процессы: Непосредственно занимаются обработкой HTTP-запросов. По умолчанию их количество соответствует числу ядер процессора.
Cache manager / loader: Вспомогательные процессы, которые следят за лимитами кэша и удаляют устаревшие данные.
Конфигурация и управление
Основной файл: Конфигурация описывается в файле nginx.conf (обычно расположен в /etc/nginx/nginx.conf).
Виртуальные серверы: NGINX позволяет настраивать множество виртуальных серверов на одном IP-адресе, разделяя запросы по доменным именам.
Команды управления:
nginx -t — проверка синтаксиса конфигурационного файла.
nginx -s reload — перезапуск сервера для применения изменений.
Модули и интеграция
NGINX является расширяемым сервером и поддерживает взаимодействие с различными технологиями:
Сетевые сокеты (FastCGI / SCGI): Для связи с приложениями, написанными на C/C++, C#, Go.
uWSGI: Специализированный протокол для Python-фреймворков (Django, Flask).
SSL/TLS: Поддержка шифрования трафика (HTTPS) и работа с сертификатами X.509.
WebDAV: С помощью дополнительных модулей NGINX может выступать в роли сервера для совместной работы с файлами, поддерживая создание папок, загрузку и удаление объектов.
Журналы (Логи) NGINX
Логи — это специальные файлы, в которые NGINX записывает информацию о своей работе. Они используются для мониторинга сервера, поиска ошибок, анализа нагрузки и расследования инцидентов безопасности.
1. Access Log (Журнал доступа)
Содержит информацию о каждом обработанном запросе.
В журнал обычно записываются:
IP-адрес клиента;
дата и время запроса;
HTTP-метод (GET, POST, PUT и др.);
запрошенный URL;
код ответа сервера (200, 404, 500 и др.);
объем переданных данных;
браузер или клиентское приложение.
Пример записи:
192.168.1.10 - - [06/Jun/2026:12:30:15 +0300] "GET /index.html HTTP/1.1" 200 1250
По access.log можно определить:
кто обращался к серверу;
какие страницы посещались;
какие API вызываются чаще всего;
наличие подозрительной активности или атак.
2. Error Log (Журнал ошибок)
Содержит информацию об ошибках и сбоях в работе сервера.
В журнал записываются:
ошибки конфигурации;
недоступность серверов приложений;
проблемы с SSL-сертификатами;
ошибки подключения к базе данных через приложение;
внутренние ошибки NGINX.
Пример записи:
2026/06/06 12:35:22 [error] 1234#0: *15 connect() failed (111: Connection refused)
По error.log администратор может быстро определить причину неисправности системы.
Развертывание
1. Традиционное развертывание (в ОС Linux)
При установке напрямую в систему NGINX функционирует через систему процессов: один master-процесс(читает конфигурацию) и несколько worker-процессов (обрабатывают запросы).
Конфигурирование: Основной файл настроек находится по пути /etc/nginx/nginx.conf. Перед запуском обязательно выполняется проверка синтаксиса командой nginx -t.
Статический контент: NGINX часто используется для размещения фронтенд-приложений (например, React-bundle). Файлы размещаются в директориях на сервере, к которым прописывается путь в блоке server конфигурации.
Права доступа: Важно назначить правильного владельца папок (например, пользователя www-data) и установить необходимые привилегии (chmod) для корректной работы сервера с файлами.
2. Развертывание через Docker
Этот метод позволяет упаковать NGINX со всеми настройками в изолированный образ.
Создание образа: Можно использовать официальный образ или создать свой на основе Dockerfile. В Dockerfile можно описать копирование ваших конфигов и статики внутрь образа.
Запуск контейнера: Для запуска используется команда docker run с параметрами:
-d: запуск в фоновом режиме.
-p: проброс портов (например, -p 80:80), чтобы приложение было доступно извне.
-v: монтирование папок с хоста в контейнер для передачи конфигурационных файлов или статики без пересборки образа.
Проверка: Внутри контейнера также можно распечатать текущую конфигурацию или проверить её синтаксис.
3. Оркестрация через Docker Compose
Для развертывания NGINX в связке с другими сервисами (например, бэкендом или БД) используется Docker Compose.
Секция services: В YAML-файле описывается сервис NGINX, указываются порты (ports), тома (volumes) и зависимости (depends_on).
Взаимодействие: NGINX может выступать как обратный прокси (reverse proxy), перенаправляя запросы к другим контейнерам внутри сети user-defined bridge.
4. Специальные сценарии развертывания
HTTPS: Для работы по защищенному протоколу необходимо сгенерировать сертификаты X.509 (например, через OpenSSL), разместить сертификат ресурса и его секретный ключ в директории приложения и настроить виртуальный сервер в NGINX на порт 443.
WebDAV: Если NGINX разворачивается как файловый сервер WebDAV, требуются дополнительные модули, создание пользователей через пакеты аутентификации и специфическая настройка методов (PUT, DELETE, MKCOL и др.) в конфигурационном файле.
При любом способе развертывания рекомендуется использовать NGINX в качестве Edge Layer (крайнего уровня) системы для обеспечения централизованной безопасности и балансировки нагрузки.



3. NGINX: проксирование http-запросов, конфигурирование прокси-сервера.
1. Проксирование HTTP-запросов
Проксирование — это процесс, при котором NGINX принимает запрос от клиента и перенаправляет его другому серверу. Клиент взаимодействует только с NGINX и не знает о существовании внутренних серверов приложения.
Reverse Proxy (обратный прокси) — принимает запросы пользователей и передает их внутренним сервисам. Используется для повышения безопасности, балансировки нагрузки и централизованного управления трафиком.
Forward Proxy (прямой прокси) — действует от имени клиента и скрывает его IP-адрес при обращении к внешним ресурсам.
API Gateway — расширенная роль обратного прокси, при которой NGINX становится единой точкой входа в систему и выполняет маршрутизацию запросов, проверку аутентификации, логирование и контроль трафика.
2. Конфигурирование прокси-сервера
Конфигурирование прокси-сервера NGINX выполняется через конфигурационные файлы, главным из которых является nginx.conf. В настройках описывается, какие запросы принимает NGINX, на какие внутренние серверы их перенаправляет и какие дополнительные параметры применяет.

Основная настройка выполняется внутри блока server. В нем указывается порт, доменное имя и правила обработки запросов.
server {
    listen 80;
    server_name example.com;

    location / {
        proxy_pass http://localhost:3000;
    }
}

В этом примере NGINX принимает запросы на example.com и перенаправляет их на приложение, работающее на localhost:3000.

Также можно настраивать разные маршруты для разных частей сайта. Например, статические файлы NGINX отдает сам, а API-запросы передает бэкенду:
server {
    listen 80;
    server_name example.com;
    location / {
        root /var/www/site;
    }
    location /api/ {
        proxy_pass http://localhost:3000;
    }
}
Здесь запросы к / обрабатываются как статические файлы, а запросы к /api/ перенаправляются на сервер приложения.

Для передачи данных о клиенте часто добавляют заголовки:
proxy_set_header Host $host;
proxy_set_header X-Real-IP $remote_addr;
proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;

Они нужны, чтобы внутреннее приложение понимало, с какого адреса пришел пользователь и через какой хост был выполнен запрос.

Если приложение запущено в нескольких экземплярах, можно настроить балансировку нагрузки:
upstream backend {
    server localhost:3000;
    server localhost:3001;
}

server {
    listen 80;

    location /api/ {
        proxy_pass http://backend;
    }
}

В этом случае NGINX будет распределять запросы между двумя серверами приложения.

Для HTTPS дополнительно указываются SSL-сертификаты:
server {
    listen 443 ssl;
    server_name example.com;

    ssl_certificate /etc/ssl/cert.pem;
    ssl_certificate_key /etc/ssl/key.pem;
}

После изменения настроек конфигурацию проверяют командой nginx -t. Если ошибок нет, изменения применяют командой nginx -s reload.
3. Практическое применение проксирования
В современных веб-системах NGINX обычно располагается перед приложением:
Клиент → NGINX → ASP.NET / Node.js / Go → БД
При этом он выполняет следующие функции:
принимает все входящие HTTP/HTTPS-запросы;
отдает статические файлы (HTML, CSS, JavaScript, изображения);
перенаправляет динамические запросы к приложению;
распределяет нагрузку между несколькими серверами;
выполняет SSL/TLS-терминацию и обрабатывает HTTPS-соединения;
скрывает внутреннюю структуру системы от внешних пользователей.
4. Архитектура NGINX
NGINX использует многопроцессную архитектуру.
Master-процесс управляет конфигурацией и рабочими процессами.
Worker-процессы непосредственно обрабатывают клиентские запросы и выполняют проксирование.
Такая архитектура обеспечивает высокую производительность и позволяет обслуживать большое количество одновременных соединений.


4. NGINX: статические файлы, конфигурирование доступа к статическим файлам.
1. Статические файлы
Статические файлы — это файлы, которые сервер отдает клиенту без изменения содержимого. К ним относятся HTML-страницы, CSS-файлы, JavaScript, изображения, аудио, видео и другие ресурсы сайта.
Доступ к таким файлам осуществляется по HTTP-запросу, после чего сервер просто считывает файл с диска и отправляет его пользователю.
2. Роль NGINX при работе со статикой
NGINX является высокопроизводительным веб-сервером и эффективно обслуживает статические файлы. При запросе изображения, CSS или JavaScript-файла он отдает его напрямую, без участия сервера приложения.
Это позволяет:
уменьшить нагрузку на сервер приложения;
ускорить загрузку сайта;
повысить производительность системы;
эффективнее использовать ресурсы сервера.
Например, NGINX может напрямую отдавать React-bundle, изображения и стили, а к приложению перенаправлять только API-запросы.
3. Конфигурирование доступа к статическим файлам
Настройка доступа к статическим файлам выполняется в конфигурационных файлах NGINX. Администратор указывает каталог на диске, из которого необходимо отдавать файлы пользователям.
Например:
server {
    listen 80;
    server_name mysite.com;

    location / {
        root /var/www/site;
    }
}
В данном примере при запросе:
http://mysite.com/index.html
NGINX найдет файл:
/var/www/site/index.html
и отправит его пользователю.

Также можно выделять отдельные каталоги для различных типов ресурсов:
location /images/ {
    root /var/www/site;
}
Тогда запрос:
http://mysite.com/images/logo.png
будет направлен к файлу:
/var/www/site/images/logo.png

Для повышения производительности часто настраивают кэширование статических файлов:
location /assets/ {
    root /var/www/site;
    expires 30d;
}
В этом случае браузер сможет хранить файлы 30 дней и не запрашивать их повторно при каждом открытии сайта.

Конфигурирование доступа к статическим файлам в NGINX заключается в указании каталогов с файлами, настройке правил обработки запросов и параметров кэширования. При получении запроса сервер находит соответствующий файл на диске и отдает его пользователю без участия сервера приложения. Это обеспечивает высокую производительность и снижает нагрузку на систему.
4. Преимущества использования NGINX для статики
высокая скорость обработки запросов;
низкое потребление памяти и процессорного времени;
возможность обслуживания большого количества пользователей одновременно;
разгрузка серверов приложений (ASP.NET Core, Node.js, Go, Java и др.).
Типичная схема работы выглядит следующим образом:
Клиент
     ↓
 NGINX
 ↙     ↘
Статика   	API
(CSS, JS, картинки)  	    ↓
 Приложение
Статические файлы NGINX отдает самостоятельно, а динамические запросы перенаправляет приложению.





5. NGINX: применение в SPA-приложениях. 
1. Что такое SPA
SPA (Single Page Application) — это одностраничное веб-приложение, в котором пользователь загружает одну HTML-страницу, а дальнейшее взаимодействие происходит через JavaScript без полной перезагрузки сайта. Примерами SPA являются приложения на React, Vue и Angular.
После сборки SPA превращается в набор статических файлов:
index.html
JavaScript-файлы (bundle)
CSS-файлы
изображения и другие ресурсы
2. Роль NGINX в SPA
Основная задача NGINX — размещение и отдача статических файлов SPA пользователям.
Например, после сборки React-приложения появляется папка:
dist/
 ├── index.html
 ├── assets/main.js
 ├── assets/main.css
NGINX настраивается так, чтобы отдавать эти файлы напрямую:
server {
    listen 80;
    server_name site.com;

    root /var/www/dist;
}
Когда пользователь открывает сайт, NGINX отправляет ему index.html, CSS и JavaScript-файлы без участия сервера приложения.
3. Работа с API через Reverse Proxy
Сам фронтенд не хранит данные, поэтому ему необходимо обращаться к серверному API.
Типичная схема выглядит так:
Пользователь
      ↓
    NGINX
   ↙     ↘
 SPA      API
(статика) (Node.js, ASP.NET, Go)

Запросы к файлам:
/main.js
/style.css

обрабатывает сам NGINX.
Запросы к API:
/api/users
/api/orders

NGINX перенаправляет на сервер приложения.
Например:
location /api/ {
    proxy_pass http://backend:3000;
}
Это позволяет скрыть внутренние адреса серверов и использовать единый адрес для всего приложения.
4. HTTPS и безопасность
В большинстве SPA именно NGINX принимает HTTPS-соединения и работает с SSL/TLS-сертификатами.
Пользователь устанавливает защищенное соединение с NGINX, после чего сервер может передавать запросы внутренним сервисам. Это упрощает настройку безопасности и снижает нагрузку на приложение.
5. Производительность и масштабируемость
NGINX способен обслуживать большое количество пользователей одновременно благодаря архитектуре Master/Worker.
Если сервер приложения запущен в нескольких экземплярах, NGINX может выполнять балансировку нагрузки и распределять запросы между ними.
Например:
           		   NGINX
          ↙   ↓   ↘
      API1  API2  API3
Это повышает производительность и отказоустойчивость системы.
Краткий вывод
В SPA-приложениях NGINX используется как сервер статических файлов и обратный прокси. Он отдает пользователям HTML, CSS и JavaScript-файлы, перенаправляет запросы к API на сервер приложения, обеспечивает работу HTTPS и при необходимости выполняет балансировку нагрузки между несколькими экземплярами бэкенда.





6. NGINX: конфигурирование виртуального сервиса для HTTPS
1. Назначение HTTPS и роль NGINX
HTTPS (HyperText Transfer Protocol Secure) — это защищенная версия протокола HTTP, использующая криптографический протокол TLS для обеспечения конфиденциальности, целостности данных и аутентификации сервера.
В современных веб-системах NGINX часто выступает в роли точки SSL/TLS-терминации. Он принимает HTTPS-запросы от клиентов, выполняет операции шифрования и дешифрования, а затем передает запросы внутренним сервисам. Это снижает нагрузку на серверы приложений и централизует настройку безопасности.
2. Подготовка сертификатов
Для настройки HTTPS необходимы:
SSL/TLS-сертификат стандарта X.509;
закрытый (секретный) ключ сервера.
Сертификат подтверждает подлинность сайта и содержит открытый ключ, информацию о владельце и данные центра сертификации. Закрытый ключ хранится только на сервере и используется для установления защищенного соединения.
Сертификаты могут быть получены двумя способами:
От доверенного центра сертификации (CA) — используется для реальных сайтов в Интернете. Браузеры автоматически доверяют таким сертификатам и отображают защищенное соединение.
Самоподписанные сертификаты (Self-Signed) — создаются администратором самостоятельно и обычно применяются в локальной разработке, тестовых средах и внутренних корпоративных системах.
Процесс подготовки сертификатов обычно включает генерацию закрытого ключа, создание запроса на сертификат (CSR) и получение готового сертификата от центра сертификации либо его самостоятельное создание с помощью утилит OpenSSL или mkcert.
3. Конфигурирование виртуального сервера
Настройка выполняется в файле nginx.conf или подключаемых конфигурационных файлах.
Для работы HTTPS создается виртуальный сервер (server), в котором указываются:
порт 443 для HTTPS-соединений;
доменное имя сайта;
путь к сертификату;
путь к закрытому ключу.
Пример конфигурации:
server {
    listen 443 ssl;
    server_name example.com;

    ssl_certificate /etc/nginx/certs/site.crt;
    ssl_certificate_key /etc/nginx/certs/site.key;

    location / {
        proxy_pass http://localhost:3000;
    }
}
В данной конфигурации NGINX принимает HTTPS-запросы на порту 443, использует указанный сертификат и ключ, а затем перенаправляет запросы внутреннему приложению.
4. Настройка параметров TLS
NGINX позволяет дополнительно настраивать параметры безопасности соединения:
используемые версии TLS (например TLS 1.2 и TLS 1.3);
наборы криптографических алгоритмов (Cipher Suites);
параметры проверки сертификатов;
политики безопасности HTTPS.
Например:
server {
   listen 443 ssl;
   server_name example.com;

   ssl_certificate /etc/nginx/certs/site.crt;
   ssl_certificate_key /etc/nginx/certs/site.key;

   ssl_protocols TLSv1.2 TLSv1.3;
   ssl_ciphers HIGH:!aNULL:!MD5;
}
В данной конфигурации директива ssl_protocols определяет допустимые версии TLS, а директива ssl_ciphers задает набор разрешенных криптографических алгоритмов для защищенного соединения.
Использование современных версий TLS обеспечивает надежную защиту данных от перехвата и подмены.
5. SSL/TLS-терминация
Одной из важных функций NGINX является SSL/TLS-терминация.
SSL/TLS-терминация — это процесс, при котором NGINX принимает HTTPS-соединение от клиента, выполняет операции шифрования и дешифрования данных, а затем передает запрос внутреннему приложению по обычному HTTP. Это упрощает настройку безопасности и снижает нагрузку на серверы приложений.

Схема работы:
Клиент
    ↓ HTTPS
  NGINX
    ↓ HTTP
Приложение
NGINX принимает зашифрованное соединение, выполняет дешифрование данных и передает запрос внутреннему сервису. Благодаря этому сервер приложения освобождается от выполнения криптографических операций и может сосредоточиться на бизнес-логике.
6. Проверка и применение конфигурации
После внесения изменений конфигурацию необходимо проверить:
nginx -t — проверка синтаксиса конфигурационных файлов;
nginx -s reload — применение изменений без остановки сервера.
Это позволяет безопасно обновлять настройки и избегать ошибок при запуске.





7. API Gateway: назначение, применение, реализации
1. Определение API Gateway
API Gateway (шлюз API) — это архитектурный паттерн, представляющий собой единую точку входа для всех клиентских запросов в распределенной системе. Вместо прямого обращения к отдельным микросервисам клиент взаимодействует только со шлюзом, который принимает запросы, обрабатывает их и перенаправляет нужным сервисам.
Схема работы:
      Клиент
         ↓
    API Gateway
   ↙     ↓    ↘
Users Orders Payments
Вместо множества адресов сервисов клиент использует один адрес шлюза, а маршрутизацией занимается API Gateway.
2. Назначение и преимущества
Основное назначение API Gateway заключается в централизации управления системой и упрощении взаимодействия клиентов с микросервисной архитектурой.
Преимущества использования:
Единая точка входа — все запросы проходят через один компонент.
Сокрытие внутренней инфраструктуры — клиенты не знают реальные IP-адреса и порты сервисов.
Централизованная безопасность — проверка JWT-токенов, OAuth2 и других механизмов аутентификации выполняется в одном месте.
Упрощение развития системы — внутреннюю структуру сервисов можно изменять без изменения клиентских приложений.
Поддержка различных клиентов — веб-приложений, мобильных приложений, десктопных программ и других систем.
3. Основные функции API Gateway
Маршрутизация (Routing)
Шлюз определяет, какому сервису необходимо передать запрос.
Например:
/api/users     → User Service
/api/orders    → Order Service
/api/payments  → Payment Service
Аутентификация и авторизация
API Gateway может централизованно проверять:
JWT-токены;
OAuth2-авторизацию;
данные Identity-серверов.
Это позволяет не дублировать одинаковую проверку безопасности во всех сервисах.
Балансировка нагрузки (Load Balancing)
Если сервис запущен в нескольких экземплярах, шлюз распределяет запросы между ними.
Это повышает производительность и отказоустойчивость системы.
Агрегация запросов
Шлюз может обращаться сразу к нескольким сервисам и объединять результаты в один ответ.
Например, при открытии профиля пользователя шлюз может одновременно получить данные пользователя, список заказов и информацию о платежах, а затем вернуть клиенту единый ответ.
Ограничение запросов (Rate Limiting)
Позволяет ограничивать количество обращений к API за определенный промежуток времени.
Например:
не более 100 запросов в минуту от одного пользователя;
защита от перегрузок и DDoS-атак.
Логирование и мониторинг
Шлюз ведет централизованный сбор информации о запросах:
журналирует обращения пользователей;
фиксирует ошибки;
собирает статистику производительности;
помогает отслеживать состояние системы.
QoS и Circuit Breaker
Для повышения надежности используются механизмы качества обслуживания (QoS).
Наиболее известный механизм — Circuit Breaker (прерыватель цепи).
Если сервис временно недоступен, шлюз перестает отправлять ему запросы и перенаправляет трафик на работоспособные узлы либо возвращает контролируемую ошибку. Это предотвращает перегрузку системы и дает сервису время на восстановление.
4. Backend for Frontend (BFF)
API Gateway может реализовывать паттерн Backend for Frontend (BFF).
Суть подхода заключается в создании отдельных API для разных типов клиентов:
веб-приложения;
мобильного приложения;
Smart TV;
других устройств.
Каждый клиент получает только необходимые ему данные и не зависит от особенностей других платформ.
5. Реализации API Gateway
Инфраструктурный уровень (Edge Layer)
Наиболее популярные решения:
NGINX;
HAProxy;
Apache HTTP Server.
Такие решения работают на уровне инфраструктуры и принимают все внешние запросы к системе.
Уровень приложений (.NET)
Для платформы .NET широко используются:
Ocelot;
YARP (Yet Another Reverse Proxy).
Они интегрируются непосредственно в ASP.NET Core-приложения и позволяют реализовывать полноценный API Gateway средствами платформы.
Облачные решения
В облачных средах применяются готовые сервисы:
Azure API Management;
AWS API Gateway;
инструменты маршрутизации Kubernetes.
Они предоставляют готовые механизмы безопасности, мониторинга, балансировки нагрузки и масштабирования.
6. Отличие API Gateway от Load Balancer
Эти понятия часто путают.
Load Balancer распределяет запросы между несколькими экземплярами одного сервиса для повышения производительности и отказоустойчивости.
API Gateway является единой точкой входа и помимо балансировки выполняет маршрутизацию, аутентификацию, логирование, ограничение запросов и другие функции управления трафиком.
Как это выглядит в реализации
Создается отдельное приложение или настраивается готовый сервер
NGINX
Ocelot
YARP
Kong
AWS API Gateway
Azure API Management
В нем описываются маршруты
/api/users    → UserService
/api/orders   → OrderService
/api/payments → PaymentService
Gateway принимает запрос клиента
Например:
GET /api/users/5
Gateway определяет нужный сервис
/api/users/5 → UserService
Gateway пересылает запрос внутреннему сервису
API Gateway → http://user-service:5001/users/5
Сервис возвращает ответ обратно через Gateway
UserService → API Gateway → Клиент
Пример на NGINX
server {
    listen 80;
    server_name api.example.com;

    location /api/users/ {
        proxy_pass http://user-service:5001/;
    }

    location /api/orders/ {
        proxy_pass http://order-service:5002/;
    }

    location /api/payments/ {
        proxy_pass http://payment-service:5003/;
    }
}
Здесь NGINX работает как API Gateway: он принимает все запросы и по пути URL решает, в какой сервис их отправить.




8. Сервер Ocelot: построение API Gateway, основные возможности
1. Что такое Ocelot и для чего он нужен
Ocelot — это легкий фреймворк для платформы ASP.NET Core, предназначенный для создания API Gateway в микросервисной архитектуре. Он распространяется с открытым исходным кодом и является одним из самых популярных решений для построения шлюзов приложений в экосистеме .NET. Основным конкурентом Ocelot считается YARP (Yet Another Reverse Proxy) от Microsoft.
API Gateway на базе Ocelot выступает единой точкой входа для всех клиентских запросов. Вместо того чтобы обращаться напрямую к десяткам микросервисов, клиент взаимодействует только со шлюзом.
Схема работы:
 Клиент
  	    ↓
 	 Ocelot
  ↙      ↓      ↘
Users Orders Payments
Такой подход дает несколько преимуществ:
скрывает реальные адреса и порты внутренних сервисов;
централизует аутентификацию и авторизацию;
упрощает изменение внутренней архитектуры системы;
позволяет контролировать весь входящий трафик в одном месте;
облегчает поддержку и развитие микросервисной системы.
2. Построение API Gateway на базе Ocelot
Для создания шлюза необходимо создать отдельное ASP.NET Core-приложение и подключить пакет Ocelot через NuGet.
После установки основная логика работы шлюза описывается в файле ocelot.json, где задаются маршруты и правила обработки запросов.
Например:
{
  "Routes": [
    {
      "UpstreamPathTemplate": "/api/users/{everything}",
      "DownstreamPathTemplate": "/api/users/{everything}",
      "DownstreamHostAndPorts": [
        {
          "Host": "userservice",
          "Port": 5001
        }
      ]
    }
  ]
}
Здесь Ocelot принимает запросы:
/api/users/*
и автоматически перенаправляет их в сервис пользователей.
В файле Program.cs подключается конфигурация Ocelot и добавляется middleware шлюза в конвейер обработки запросов.
После запуска приложение начинает выполнять функции полноценного API Gateway.
3. Маршрутизация и агрегация запросов
Основной задачей Ocelot является маршрутизация (Routing).
Шлюз анализирует входящий URL и определяет, какому сервису необходимо передать запрос.
Например:
/api/users    → User Service
/api/orders   → Order Service
/api/payments → Payment Service
Кроме обычной маршрутизации Ocelot поддерживает агрегацию запросов.
В этом режиме шлюз может обратиться сразу к нескольким микросервисам, объединить полученные данные и вернуть клиенту единый ответ.
Например, при открытии профиля пользователя могут одновременно вызываться:
сервис пользователей;
сервис заказов;
сервис платежей.
После чего клиент получает один объединенный ответ вместо нескольких отдельных запросов.
4. Безопасность и аутентификация
Одним из главных преимуществ API Gateway является централизованная безопасность.
Ocelot поддерживает:
JWT-аутентификацию;
OAuth2;
OpenID Connect;
интеграцию с серверами идентификации (например, OpenIddict).
Схема работы выглядит следующим образом:
Клиент
    ↓ JWT
  Ocelot
    ↓
Микросервис
Шлюз сначала проверяет токен пользователя и только после успешной проверки передает запрос внутреннему сервису.
Благодаря этому каждый микросервис не обязан самостоятельно реализовывать одинаковую логику проверки безопасности.
5. Балансировка нагрузки
Если сервис запущен в нескольких экземплярах, Ocelot способен распределять между ними нагрузку.
Поддерживаются различные алгоритмы балансировки:
Round Robin
Запросы распределяются по очереди между серверами.
Запрос 1 → Server1
Запрос 2 → Server2
Запрос 3 → Server3
Least Connection
Запрос направляется серверу с наименьшим количеством активных соединений.
Cookie Sticky Session
Пользователь закрепляется за определенным сервером на основе Cookie, что позволяет сохранять состояние сессии.
Также разработчик может реализовать собственные алгоритмы балансировки.
6. Отказоустойчивость и QoS
Для повышения надежности системы Ocelot поддерживает механизмы Quality of Service (QoS).
Наиболее важным является Circuit Breaker (прерыватель цепи).
Если сервис начинает работать нестабильно или перестает отвечать, шлюз временно прекращает отправлять ему запросы.
Схема:
Service ❌
    ↑
Circuit Breaker

Это предотвращает перегрузку системы и дает сервису возможность восстановиться.
Дополнительно поддерживается:
Service Discovery
Автоматический поиск доступных экземпляров сервисов через специальные реестры, например Consul.
Rate Limiting
Ограничение количества запросов от пользователя за определенный промежуток времени.
Например:
Не более 100 запросов в минуту
Это защищает систему от перегрузки и злоупотреблений API.
7. Расширяемость и мониторинг
Ocelot предоставляет дополнительные механизмы расширения функциональности.
Delegating Handlers
Специальные обработчики, которые позволяют выполнять дополнительный код перед отправкой запроса сервису.
С их помощью можно реализовывать:
дополнительную проверку данных;
логирование;
изменение заголовков запросов;
собственные механизмы безопасности.
Логирование (Logging)
Шлюз может централизованно фиксировать:
входящие запросы;
ошибки;
время обработки;
состояние сервисов.
Кэширование
Ocelot поддерживает сохранение результатов запросов для ускорения повторных обращений к одним и тем же данным.
Это снижает нагрузку на внутренние сервисы и повышает производительность системы.

API Gateway — это архитектурный паттерн единой точки входа в систему, а Ocelot — его конкретная реализация для платформы ASP.NET Core, позволяющая создавать шлюзы приложений с маршрутизацией, аутентификацией, балансировкой нагрузки и другими функциями управления трафиком.





9. Создание сервиса (daemon) Ocelot API Gateway в Linux
1. Назначение службы (daemon)
После разработки и настройки API Gateway на базе Ocelot его необходимо постоянно поддерживать в рабочем состоянии. Для этого в Linux приложение обычно запускается не вручную из терминала, а в виде системной службы (daemon).
Daemon — это фоновый процесс, который запускается вместе с операционной системой и продолжает работать независимо от пользователя. Благодаря этому Ocelot автоматически стартует после перезагрузки сервера, может перезапускаться при сбоях и централизованно управляться средствами операционной системы.
2. Подготовка приложения к развертыванию
Перед созданием службы необходимо собрать приложение для Linux.
Для Ocelot обычно используется self-contained публикация, при которой все необходимые компоненты .NET Runtime включаются в итоговый файл.
Преимущества такого подхода:
на сервере не требуется установка .NET Runtime;
приложение переносится одним файлом;
упрощается развертывание и обновление.
При публикации указывается целевая платформа, например:
linux-x64
После сборки получается исполняемый файл API Gateway.
3. Копирование приложения на сервер
Полученный файл необходимо перенести на Linux-сервер.
Для этого часто используется утилита:
scp
Она позволяет безопасно копировать файлы по протоколу SSH.
После переноса приложение обычно размещается в отдельной директории, например:
/opt/ocelot/
или

4. Настройка прав доступа

chmod +x OcelotGateway
Она добавляет право на выполнение программы.
Также при необходимости назначаются владелец файла и права доступа, чтобы ограничить возможность запуска или изменения приложения посторонними пользователями.
5. Создание службы systemd
В современных дистрибутивах Linux управление службами выполняет система systemd.

/etc/systemd/system/ocelot.service

6. Управление службой

10. Балансировщики запросов: определение, назначение, применение, реализации
1. Что такое балансировщик нагрузки
Балансировщик нагрузки (Load Balancer) — это программный или аппаратный компонент системы, который распределяет входящие запросы между несколькими экземплярами серверов или сервисов по определенным правилам.
Без балансировщика все запросы поступают на один сервер:
Клиенты
    ↓
 Server 1
Если сервер перегружается или выходит из строя, система становится недоступной.
С балансировщиком схема выглядит следующим образом:
Клиенты
    ↓
Load Balancer
 ↙   ↓   ↘
S1   S2   S3
Балансировщик принимает запросы и решает, на какой сервер их отправить.
2. Назначение балансировщика
Основная задача балансировщика — обеспечить эффективное использование ресурсов системы и повысить ее надежность.
Основные преимущества:
Повышение отказоустойчивости
Если один сервер перестает отвечать, балансировщик автоматически исключает его из работы и перенаправляет запросы на остальные узлы.
Server 2 ❌


Запросы → Server 1 и Server 3
Пользователи при этом могут даже не заметить отказ.
Повышение производительности
Нагрузка распределяется между несколькими серверами, поэтому система способна обслуживать значительно больше пользователей одновременно.
Например, три сервера способны обработать примерно в три раза больше запросов, чем один.
Равномерное распределение нагрузки
Балансировщик предотвращает ситуацию, когда один сервер перегружен, а остальные простаивают без работы.
3. Основные алгоритмы балансировки
Способ распределения запросов определяется выбранным алгоритмом.
Round Robin
Самый простой и популярный алгоритм.
Запросы распределяются по очереди:
Запрос 1 → Server 1
Запрос 2 → Server 2
Запрос 3 → Server 3
Запрос 4 → Server 1
Все серверы получают одинаковое количество запросов.
Weighted Round Robin
Используется, когда серверы имеют разную производительность.
Например:
Server 1 (вес 3)
Server 2 (вес 1)

Тогда Server 1 будет получать примерно в три раза больше запросов.
Least Connection
Запрос направляется серверу с наименьшим количеством активных соединений.
Например:
Server 1 → 50 клиентов
Server 2 → 10 клиентов
Новый запрос получит Server 2.
Алгоритм особенно эффективен, когда запросы имеют разную продолжительность выполнения.
Weighted Least Connection
Учитывает одновременно:
количество активных соединений;
производительность сервера.
Используется в больших кластерах с неоднородным оборудованием.
IP Hash
Балансировщик вычисляет хеш по IP-адресу клиента и всегда отправляет его на один и тот же сервер.
Клиент A → Server 1
Клиент B → Server 2

Это удобно для хранения пользовательских данных на конкретном узле.
Sticky Session (Session Affinity)
После первого обращения пользователь закрепляется за определенным сервером.
Пользователь
      ↓
  Server 2
      ↓
Все последующие запросы
      ↓
  Server 2

Часто используется, если данные сессии хранятся в памяти приложения.
Random
Сервер выбирается случайным образом.
Несмотря на простоту, при большом количестве запросов нагрузка распределяется достаточно равномерно.
Least Response Time
Балансировщик анализирует скорость ответа серверов и отправляет запрос тому узлу, который отвечает быстрее остальных.
Consistent Hashing
Используется в распределенных системах хранения данных и кэширования.
Позволяет минимизировать перераспределение данных при добавлении или удалении серверов.
4. Реализации балансировщиков
Балансировка нагрузки может реализовываться различными средствами.
Инфраструктурный уровень
Наиболее популярные решения:
NGINX;
HAProxy;
Apache HTTP Server.
Например, NGINX может одновременно работать как:
веб-сервер;
обратный прокси;
API Gateway;
балансировщик нагрузки.
Уровень приложений
В экосистеме .NET балансировка может выполняться через Ocelot.
Поддерживаются алгоритмы:
Round Robin;
Least Connection;
Cookie Sticky Session;
собственные пользовательские балансировщики.
Облачные решения
Облачные провайдеры предоставляют готовые балансировщики:
Azure Load Balancer;
AWS Elastic Load Balancer;
Google Cloud Load Balancer.
Преимущество таких решений — автоматическое масштабирование и минимальные требования к настройке.
Kubernetes
В Kubernetes балансировка встроена в инфраструктуру контейнеров.
Трафик автоматически распределяется между несколькими экземплярами приложения (Pods).
5. Health Checking и Service Discovery
Современный балансировщик не просто распределяет запросы.
Он должен постоянно проверять работоспособность серверов.
Health Checking
Балансировщик регулярно отправляет тестовые запросы сервисам.
Если сервер не отвечает:
Server 2 ❌
он временно исключается из списка доступных узлов.
После восстановления сервер автоматически возвращается в работу.
Service Discovery
В динамических средах адреса сервисов могут постоянно меняться.
Для решения этой проблемы используются системы регистрации сервисов:
Consul;
etcd;
Kubernetes Service Discovery.
Балансировщик получает от них актуальную информацию о доступных экземплярах приложений.
6. QoS и отказоустойчивость
Для предотвращения каскадных сбоев используются механизмы Quality of Service (QoS).
Наиболее известный механизм — Circuit Breaker (Прерыватель цепи).
Если сервис начинает работать нестабильно, балансировщик или API Gateway временно прекращает отправку ему запросов.
Service ❌
    ↑
Circuit Breaker

Это позволяет избежать перегрузки системы и дает сервису время на восстановление.




11. Основные алгоритмы балансировки: Round Robin, Weighted Round Robin, IP Hash, StickySession.  
Балансировщик нагрузки (Load Balancer) — это компонент веб-системы (зачастую встроенный в API Gateway), предназначенный для распределения входящих запросов между несколькими экземплярами серверов. Его использование позволяет повысить отказоустойчивость, улучшить производительность и обеспечить равномерную загрузку узлов системы.
Ниже приведены определения и описание основных алгоритмов балансировки на основе источников:
1. Round Robin (Циклический перебор)
Определение: Алгоритм, распределяющий запросы равномерно по их количеству в порядке простой очереди.
Принцип работы: Первый запрос отправляется на сервер №1, второй — на сервер №2, и так далее по кругу.
Реализация: Является одним из стандартных встроенных алгоритмов в шлюзе Ocelot. Он оптимален в ситуациях, когда все серверы в кластере имеют одинаковую вычислительную мощность.
2. Weighted Round Robin (Взвешенный циклический перебор)
Определение: Модификация Round Robin, которая учитывает разную мощность и производительностьузлов кластера.
Принцип работы: Каждому серверу назначается «вес» (коэффициент). Серверы с более высоким весом получают пропорционально большее количество запросов.
Пример: Если сервер А в два раза мощнее сервера Б, ему назначается двойной вес, и он будет обрабатывать два запроса на каждый один запрос сервера Б.
3. IP Hash (Source IP Hash)
Определение: Алгоритм, обеспечивающий «прилипание» (affinity) клиента к конкретному серверу на основе его IP-адреса.
Принцип работы: Хеш-функция вычисляется от IP-адреса отправителя, и результат определяет, на какой сервер будет направлен запрос.
Назначение: Это гарантирует, что все запросы от одного и того же пользователя (с одного IP) будут попадать на один и тот же экземпляр сервера, что важно, если приложение не является полностью «stateless».
Балансировщик берет IP клиента, вычисляет от него хеш, делит результат по модулю количества серверов: 8457321 mod 4 = 1 Полученный номер соответствует серверу.
4. Sticky Sessions / Session Affinity (Прилипающие сессии)
Определение: Механизм закрепления клиента за конкретным сервером в случаях, когда сессия или состояние пользователя хранится в оперативной памяти этого сервера.
Реализация в Ocelot (Cookie Sticky Session):
Шлюз Ocelot может использовать специальные Cookie для реализации этого механизма.
Сам шлюз устанавливает set-cookie клиенту, чтобы при последующих обращениях идентифицировать его и направлять на тот же сервер, где была начата сессия.
Важность: Без этого алгоритма в системах со stateful-авторизацией пользователь может внезапно «разлогиниться», если балансировщик перебросит его запрос на другой сервер, который ничего не знает о его текущей сессии.
Дополнительные аспекты балансировки
Источники подчеркивают, что эффективная балансировка невозможна без следующих механизмов:
Интеграция с Health Checking: В реальности балансировщик должен использовать только работоспособные (healthy) узлы, автоматически исключая или понижая приоритет «сомнительных» серверов.
Least Connection: Альтернативный алгоритм (поддерживаемый Ocelot), который направляет запрос серверу с наименьшим количеством активных соединений, что эффективно при неоднородной длительности обработки запросов.
Умная балансировка (Canary): Платформы типа Consul позволяют реализовывать поэтапный переход трафика на новую версию сервиса (канареечное тестирование).





12. Сервер Ocelot: конфигурирование балансировщика запросов; разработка пользовательского балансировщика
Ocelot — это API Gateway для платформы ASP.NET Core. Он располагается между клиентом и микросервисами, принимает запросы и перенаправляет их на нужный сервис. Одной из его функций является балансировка нагрузки, то есть распределение запросов между несколькими экземплярами одного сервиса.
1. Конфигурирование балансировщика в Ocelot
Настройка выполняется в файле ocelot.json. Для каждого маршрута можно указать алгоритм балансировки, который будет использоваться при выборе сервера.
{
  "Routes": [
    {
      "UpstreamPathTemplate": "/api/users",
      "DownstreamPathTemplate": "/api/users",
      "DownstreamScheme": "http",

      "DownstreamHostAndPorts": [
        { "Host": "service1", "Port": 5000 },
        { "Host": "service2", "Port": 5000 }
      ],

      "LoadBalancerOptions": {
        "Type": "RoundRobin"
      }
    }
  ]
}
Основные встроенные алгоритмы:
Round Robin (циклический) — запросы поочередно отправляются на разные серверы. Например: Server1 → Server2 → Server3 → Server1.
Least Connection (наименьшее количество соединений) — новый запрос направляется серверу с наименьшим количеством активных подключений.
Cookie Sticky Session — пользователь закрепляется за определенным сервером через cookie. Все последующие запросы этого клиента будут попадать на тот же сервер.
"LoadBalancerOptions": {
    "Type": "LeastConnection"
}
Балансировка необходима для повышения производительности, масштабируемости и отказоустойчивости системы.
2. Пользовательский (Custom) балансировщик
Если встроенных алгоритмов недостаточно, в Ocelot можно создать собственный балансировщик.
Для этого разрабатывается класс, реализующий интерфейс балансировщика, после чего он регистрируется в приложении и указывается в конфигурации Ocelot.
Пользовательский балансировщик позволяет учитывать дополнительные параметры:
текущую загрузку процессора;
объем используемой памяти;
географическое расположение сервера;
приоритет отдельных узлов;
любые специфические требования системы.
Например, можно реализовать алгоритм, который будет направлять запросы только на серверы с загрузкой CPU менее 70%.
public class CustomLoadBalancer : ILoadBalancer
{
    public async Task<Response<ServiceHostAndPort>>
        Lease(HttpContext context)
    {
        // логика выбора сервера
    }

    public void Release(ServiceHostAndPort host)
    {
    }
}

builder.Services
    .AddOcelot()
    .AddCustomLoadBalancer<CustomLoadBalancer>();
3. Дополнительные механизмы
Для корректной работы балансировки Ocelot обычно использует дополнительные инструменты:
Health Check — проверяет доступность серверов и исключает неисправные узлы из маршрутизации.
Service Discovery (Consul) — автоматически получает список доступных экземпляров сервисов.
Circuit Breaker (QoS) — временно прекращает отправку запросов на сервис, который перестал отвечать или работает с ошибками.





13. Single Sign One в web-системах: определение, назначение, применение. 
1. Определение Single Sign-On (SSO)
Single Sign-On (SSO) (в источниках также встречается как «технология единого входа» или «Single Sign One») — это метод аутентификации, который позволяет пользователю один раз пройти процедуру проверки подлинности и получать доступ ко всем связанным ресурсам и сервисам распределенной системы без повторного ввода учетных данных.
2. Назначение SSO
Основная цель внедрения SSO — упрощение взаимодействия пользователя с распределенной инфраструктурой при сохранении высокого уровня безопасности:
Централизация управления: Идентификация пользователя происходит на одном специализированном сервере, что избавляет каждый отдельный сервис от необходимости хранить пароли пользователей.
Удобство для пользователя: Исключается необходимость запоминать и вводить разные пароли для множества микросервисов внутри одной системы.
Безопасность в распределенных системах: SSO позволяет реализовать безопасную передачу прав доступа между независимыми узлами через стандартизированные токены.
3. Основные компоненты и участники
Для функционирования SSO в системе выделяются специфические роли:
Identity-provider (поставщик удостоверений): Сервер, который проверяет личность пользователя и генерирует специальный токен доступа.
Service-provider (поставщик сервиса): Целевой сервер, предоставляющий ресурс. Он не проводит аутентификацию сам, а доверяет токену, выданному Identity-provider. Для этого оба сервера должны иметь общий секретный ключ для проверки подлинности токена.
Клиент: Может быть активным (программный код) или пассивным (браузер).
4. Реализации и протоколы
В источниках выделяются следующие технологические стандарты для построения SSO:
OpenID / OpenID Connect: Открытый стандарт, специально разработанный для реализации SSO. Работает через стандартные HTTP(S)-сообщения и позволяет децентрализованно подтверждать личность. Современная версия OpenID Connect 1.0 работает поверх протокола OAuth 2.0.
OAuth 2.0: Открытый стандарт аутентификации, который часто используется как база для SSO-решений.
Kerberos: Протокол, применяемый в интранет-сетях Windows (Active Directory), также основанный на принципе SSO.
Token-аутентификация (JWT): Является основным механизмом реализации SSO в современных веб-системах. Токен (например, JWT) содержит утверждения о пользователе (Claims) и цифровую подпись, гарантирующую его подлинность.
5. Применение в инфраструктуре (Ocelot и OpenIddict)
В контексте разработки на .NET (ASP.NET Core) для реализации SSO применяются следующие инструменты:
OpenIddict: Фреймворк с открытым кодом, предназначенный для создания собственных серверов аутентификации, поддерживающих SSO через протоколы OAuth 2.0 и OpenID Connect.
API Gateway (например, Ocelot): Выступает единой точкой входа, где централизованно проверяются SSO-токены перед тем, как запрос будет направлен внутренним сервисам.
Использование SSO позволяет строить гибкие и защищенные веб-системы, где безопасность вынесена на уровень специализированных компонентов (Identity Providers) и шлюзов (Edge Layer).
6. Как работает SSO
Последовательность действий:
Пользователь открывает приложение.
Приложение перенаправляет его на сервер авторизации.
Пользователь вводит логин и пароль.
Сервер авторизации выдает JWT-токен.
Токен передается приложению.
При обращении к другим сервисам этот токен используется повторно.
Пример JWT в запросе:
GET /api/profile
Authorization: Bearer eyJhbGciOi...
Сервис проверяет подпись токена и предоставляет доступ.



14. Токен-аутентификация: структура JWS-токена 
Токен-аутентификация представляет собой процедуру, использующую Token — битовую последовательность, построенную по определенному принципу — для идентификации пользователя или авторизации его операций. В распределенных системах этот метод является основным инструментом реализации технологии единого входа (Single Sign-On).
Одной из ключевых реализаций токенов является стандарт JWS (JSON Web Signature).
Структура JWS-токена
Согласно семейству стандартов JOSE (JavaScript Object and Encryption), предназначенных для подписи и шифрования JSON-объектов, JWS — это подписанный JSON Web Token (JWT). Его главная особенность заключается в том, что содержимое токена можно прочитать, но его нельзя подделать, что гарантирует целостность и подлинность данных.
Структурно JWS-токен состоит из трех сегментов, каждый из которых закодирован в формате BASE64URL и отделен от другого точкой:
Header (Заголовок): Содержит метаданные о токене и алгоритмах его обработки. В нем обычно указываются:
alg: используемый алгоритм подписи.
kid: идентификатор ключа.
typ: тип токена (например, at+jwt согласно RFC 908).
Payload (Полезная нагрузка): Содержит основную информацию в виде утверждений (Claims) и дополнительных инструкций (Statements). Типовой состав полезной нагрузки включает:
Issuer (эмитент): сервер, сгенерировавший токен (Identity-provider).
Audience (аудитория): целевые сервисы, для которых предназначен токен.
Expires On (время жизни): дата и время истечения действия токена.
Claim: сведения о самом пользователе.
Signature (Подпись): Третий сегмент, который представляет собой результат криптографической подписи (например, HMAC или асимметричный ключ) первых двух сегментов. Именно этот сегмент позволяет серверу-поставщику услуг (Service-provider) убедиться, что токен был выдан доверенным Identity-provider и не изменялся в процессе передачи.
Применение в инфраструктуре
В современных .NET-системах выпуск токенов в формате JWS обеспечивается фреймворками вроде OpenIddict. Такие токены передаются клиентом (активным или пассивным) шлюзу API (например, Ocelot), который централизованно проверяет подпись и структуру токена перед тем, как предоставить доступ к внутренним сервисам системы.



15. Фреймворк OpenIddict: назначение, применение, создание сервера аутентификации, интеграция с Ocelot API Gateway .    
1. Назначение и применение OpenIddict
OpenIddict — это гибкий фреймворк с открытым исходным кодом для ASP.NET Core, предназначенный для создания серверов аутентификации и идентификации.
Ключевые характеристики и возможности:
Поддержка протоколов: Обеспечивает полную поддержку стандартов OAuth 2.0 и OpenID Connect, включая OpenID Connect Discovery и OpenID Connect Core.
Типы токенов: Позволяет выпускать JWT в форматах JWS (подписанные, гарантирующие целостность) и JWE (зашифрованные, гарантирующие конфиденциальность).
Механизмы авторизации (Grants): Поддерживает основные типы разрешений:
Authorization Code (код авторизации);
Client Credentials (учетные данные клиента);
Password (учетные данные владельца ресурса);
Refresh Token (токен обновления для продления сессии).
Инфраструктура: Для хранения данных о пользователях, приложениях-клиентах и выданных токенах фреймворк использует Entity Framework Core.
2. Создание сервера аутентификации
Согласно источникам, процесс построения Identity-сервера на базе OpenIddict включает несколько этапов разработки:
Подготовка проекта: Установка необходимых NuGet-пакетов OpenIddict.
dotnet add package OpenIddict.AspNetCore
dotnet add package OpenIddict.EntityFrameworkCore
Настройка кода: Основная конфигурация сервера прописывается в файле Program.cs, где регистрируются службы OpenIddict и определяются параметры выпуска токенов.
builder.Services.AddOpenIddict()
    .AddCore(options =>
    {
        options.UseEntityFrameworkCore();
    })
    .AddServer(options =>
    {
        options.SetTokenEndpointUris("/connect/token");
        options.AllowPasswordFlow();
        options.AcceptAnonymousClients();
    });
Модель данных: Создание и настройка DbContext для интеграции OpenIddict с базой данных через Entity Framework Core.
builder.Services.AddDbContext<AppDbContext>(
    options =>
    {
        options.UseSqlServer(connectionString);
    });
Инициализация:
Проведение миграций и инициализация БД.
Программное создание первого пользователя системы.
Регистрация клиента: Определение параметров клиентского приложения (идентификатор, секрет, разрешенные области доступа/scopes).
Выдача токенов: Реализация контроллеров, отвечающих за проверку учетных данных и генерацию (выдачу) токенов клиентам.
3. Интеграция с Ocelot API Gateway
В архитектуре распределенных веб-систем OpenIddict и Ocelot работают совместно для обеспечения безопасности всей системы.
builder.Services
    .AddAuthentication("Bearer")
    .AddJwtBearer("Bearer", options =>
    {
        options.Authority =
            "https://auth.company.com";

        options.Audience = "api";
    });
Схема взаимодействия:
Централизованная точка входа: Все запросы от клиентов поступают на шлюз Ocelot.
Процесс аутентификации:
Клиент сначала обращается к серверу OpenIddict (Identity-provider) для прохождения аутентификации и получения JWT-токена.
Получив токен, клиент отправляет прикладной запрос к шлюзу Ocelot, передавая токен в заголовке.
Валидация в Ocelot: Шлюз настраивается на обработку и проверку токенов, выпущенных сервером OpenIddict.
Безопасный доступ: Если токен валиден, Ocelot проксирует запрос к внутренним микросервисам. Это позволяет скрыть адреса серверов и централизовать логику безопасности, избавляя каждый сервис от необходимости самостоятельно проверять токены.
Таким образом, OpenIddict выступает в роли Identity-provider (генератора доверия), а Ocelot — в роли Service-provider/Gateway (контролера доступа), использующего это доверие для защиты ресурсов системы.




16. Платформа Docker: техническая основа, структура платформы, репозитории, docker-образ и контейнер, стандарт Docker OCI.  
Docker представляет собой платформу для контейнеризации, которая виртуализирует операционное окружение приложения (процессы, сеть, файловую систему) в рамках хостовой ОС для его изоляции.
Техническая основа платформы
В основе Docker лежат ключевые механизмы ядра Linux:
единое ядро линукс
Namespaces (Пространства имен): функция ядра для изоляции и виртуализации глобальных системных ресурсов. Она позволяет виртуализировать:
Mount: точки монтирования файловых систем.
PID: идентификаторы процессов (внутри контейнера они свои).
Network: сетевой стек (IP, порты).
UTS: имена хоста и домена.
IPC: средства межпроцессного взаимодействия.
User: идентификаторы пользователей и групп.
Namespaces – реализовано в форме трех системных вызовов: clone (создать процесс и поместить в новые namespaces), unshare (отсоединиться о разделяемых ресурсов и получить новые namespaces); setns (присоединиться к существующему namespace).
Cgroups (Control Groups): механизм ограничения и учета системных ресурсов (процессор, память, диск, сеть) для групп процессов.
Control Groups – механизм ядра Linux, позволяющий ограничивать, учитывать и изолировать использование системных ресурсов (процессор, память, диск, сеть) для группы процессов.
OverlayFS: многослойная файловая система, которая позволяет накладывать каталоги друг на друга. Она использует слои lowerdir (только чтение) и один верхний слой upperdir (чтение и запись).
файловая система Linux, позволяет накладывать каталоги друг на друга, с единой точкой монтирования. Несколько нижних слоев (lowerdir, только чтение) и один верхний слой (upperdir, чтение и запись), одна точка подключения (merged), одинаковые слои в разных образах не дублируются. Механизм copy-up.
Copy-on-write (CoW): стратегия управления данными, при которой ресурс копируется только при попытке его изменения. В OverlayFS этот механизм называется copy-up.
Структура платформы
С точки зрения пользователя, Docker состоит из трех основных компонентов:
Клиент: интерфейс командной строки (CLI) или графическое приложение для управления.
Docker Engine: демон или сервис, выполняющий основную работу по управлению контейнерами на хосте.
Реестры (Registries): хранилища для образов.
Репозитории и реестры образов
Для хранения и распространения Docker-образов используются Container Registries.
Docker Hub: основной публичный реестр.
Альтернативы: Harbor, Azure Container Registry, Amazon Elastic Container Registry и другие.
Взаимодействие с реестрами (команды push/pull) осуществляется по специальному протоколу Distribution Specification.
Docker-образ и контейнер
Платформа разделяет понятия шаблона и работающего экземпляра:
Docker-образ (image): это шаблон контейнера, доступный только для чтения. Он содержит код приложения, переменные окружения и слои файловой системы (lowerdir). Инструкцией для его создания служит Dockerfile.
Контейнер (container): это экземпляр образа. В отличие от образа, он имеет дополнительный верхний слой (upperdir), в который разрешена запись данных во время работы приложения.
Стандарт Docker OCI (Open Container Initiative)
OCI — это проект Linux Foundation (создан в 2015 году при участии Microsoft, Google, AWS и др.), направленный на создание единых стандартов для обеспечения переносимости контейнеров между разными средами. Стандарт включает:
Runtime Specification: требования к файловой системе, конфигурации (config.json) и жизненному циклу контейнера.
Image Specification: стандарты упаковки, слоев и подписи образов.
Distribution Specification: протокол взаимодействия с реестрами.
Стандарт OCI поддерживается не только Docker, но и другими инструментами, такими как Podman, Buildah и CRI-O. Он позволяет запускать контейнеры в различных ОС: Linux, Windows (через WSL2/Hyper-V), Solaris и FreeBSD. При этом macOS напрямую с OCI-контейнерами не работает.




17. Docker: запуск и остановка контейнера, параметры запуска.  
1. Запуск контейнера
Основной командой для создания и запуска контейнера из образа является docker run. Контейнер при этом является работающим экземпляром образа (image), имеющим дополнительный верхний слой для записи (upperdir).
Основные параметры запуска (docker run):
-d (--detach): запуск контейнера в фоновом режиме (отсоединенном от терминала).
-it: запуск в интерактивном режиме с выделением псевдо-терминала (TTY). Часто используется для работы с командной оболочкой внутри контейнера (например, /bin/bash).
--name [имя]: назначение контейнеру конкретного пользовательского имени (например, tmp-hello или mssql-server) для удобства управления.
--rm: автоматическое удаление контейнера сразу после завершения его работы.
-p [хост-порт]:[контейнер-порт]: настройка соответствия портов (публикация порта наружу), что позволяет обращаться к приложению внутри контейнера через порт хостовой машины.
-v [путь-на-хосте]:[путь-в-контейнере]: монтирование тома или папки хоста в контейнер для постоянного хранения данных или передачи файлов (например, конфигураций или статики).
--network [имя-сети]: подключение контейнера к определенной сети (например, bridge по умолчанию или пользовательская сеть user-defined bridge для взаимодействия между контейнерами).
-e / environment: задание переменных окружения внутри контейнера.
2. Остановка и удаление контейнера
Для прекращения работы и очистки ресурсов используются следующие команды:
docker stop [имя/ID]: команда для корректной остановки работающего контейнера.
docker rm [имя/ID]: команда для удаления остановленного контейнера из системы.
docker rmi [имя-образа]: используется для удаления самого образа, если он больше не нужен.
3. Комплексное управление через Docker Compose
Для систем, состоящих из нескольких взаимодействующих контейнеров, используется инструмент Docker Compose, который позволяет запускать, останавливать и удалять всю группу контейнеров одной командой на основе файла конфигурации (YAML).
В секции services файла docker-compose.yml описываются те же параметры запуска в декларативном стиле:
image: указывается образ;
ports: публикация портов;
depends_on: определяет порядок запуска (зависимость одного контейнера от другого);
restart: политика автоматического перезапуска;
healthcheck: параметры проверки «здоровья» (работоспособности) приложения внутри контейнера.
4. Внутренние механизмы управления
Технически запуск контейнера обеспечивается функциями ядра Linux:
Namespaces изолируют процессы, сеть и файловую систему.
Cgroups устанавливают лимиты на использование ресурсов (CPU, RAM).
OverlayFS объединяет слои образа и временный слой контейнера в единую файловую систему.

19. Docker: принцип распространения контейнеризированных приложений 
Принцип распространения контейнеризированных приложений в Docker основан на использовании образов (images) как универсальных единиц упаковки и реестров (registries) как централизованных хранилищ для их передачи между различными узлами системы.
1. Образ как средство упаковки
Docker-образ является шаблоном контейнера, доступным только для чтения, который предназначен для упаковки и распространения приложения. Он содержит в себе:
Слои lowerdir: неизменяемые слои файловой системы.
Код приложения и его зависимости.
Переменные окружения.
Благодаря файловой системе OverlayFS, одинаковые слои в разных образах не дублируются, что значительно оптимизирует процесс их хранения и передачи по сети.
2. Реестры контейнеров (Container Registries)
Для хранения и распространения образов используются специальные службы — реестры.
Docker Hub: основной и самый популярный публичный реестр.
Другие реализации: Harbor, Azure Container Registry, Amazon Elastic Container Registry.
Взаимодействие с реестрами стандартизировано в рамках инициативы OCI (Open Container Initiative)спецификацией Distribution Specification, которая определяет протоколы для загрузки и скачивания образов.
3. Жизненный цикл распространения
Процесс передачи приложения от разработчика в целевую среду (например, на сервер Linux) состоит из нескольких этапов:
Создание образа: Образ собирается на основе инструкции Dockerfile или создается из работающего контейнера с помощью команды docker commit.
Аутентификация: Пользователь выполняет вход в реестр с помощью команды docker login.
Публикация (Push): Готовый образ загружается в репозиторий командой docker push.
Развертывание (Pull & Run): На целевом сервере выполняется команда docker run. Если образа нет локально, Docker автоматически скачивает его из реестра и запускает экземпляр приложения — контейнер.
4. Преимущества принципа распространения Docker
Переносимость: Стандарт OCI гарантирует, что образ, собранный на одной платформе, будет корректно работать в любой ОС, поддерживающей этот стандарт (различные дистрибутивы Linux, Windows через WSL2, Solaris и др.).
Изоляция: Приложение распространяется вместе со своим операционным окружением (сеть, процессы, файловая система), что исключает конфликты зависимостей на хостовой машине.
Эффективность: Использование механизма Copy-on-Write (CoW) и слоев позволяет передавать только измененные части образа, а не все приложение целиком.
Таким образом, Docker превращает процесс развертывания сложного ПО в простую операцию копирования и запуска стандартизированного шаблона, доступного в любой точке сети через реестры образов.




20. Docker: контейнеризация ASP.NET CORE-приложения.
Контейнеризация ASP.NET Core приложений в среде Docker — это метод виртуализации операционного окружения (файловой системы, переменных окружения, сети и процессов) в рамках хостовой ОС для обеспечения изоляции и переносимости приложения.
1. Подготовка приложения (Публикация)
Для успешного запуска в Docker-контейнере (обычно на базе Linux) приложение ASP.NET Core должно быть правильно собрано:
Типы сборки: Источники рекомендуют использовать форматы self-contained (автономная) или single-file (в один исполняемый файл). Это избавляет от необходимости устанавливать .NET Runtime внутри самого образа.
RID (Runtime Identifier): При создании сборки необходимо указать идентификатор целевой среды выполнения — linux-x64.
Перенос: Готовую публикацию можно скопировать на целевой сервер (где установлен Docker) с помощью утилиты scp.
Перед контейнеризацией приложение необходимо опубликовать!!!!!!!!!!!!!!!!!!!!!!
это главное отличие от GO
Например:
dotnet publish -c Release -r linux-x64
где:
Release — релизная сборка;
linux-x64 — целевая Linux-платформа.
После публикации появляется каталог:
publish/
├─ MyApi.dll
├─ appsettings.json
└─ ...
Именно эти файлы будут помещены в Docker-образ.
2. Создание Docker-образа
Образ (image) служит шаблоном для контейнера и создается на основе текстовой инструкции — Dockerfile.
Выбор базового образа: Для ASP.NET Core используются специализированные образы Linux.
Проблема «голого» (minimal) Linux: При использовании максимально облегченных образов может возникнуть ошибка: «не найдена ICU» (библиотеки для поддержки Юникода). В таких случаях требуется либо установка ICU в образ, либо использование базового образа, где эти компоненты уже присутствуют.
Сборка: Команда сборки образа выполняется на основе Dockerfile, после чего образу можно присвоить алиас (тег) для удобства версионирования.
Dockerfile содержит инструкции по сборке образа.
Простейший пример:
FROM mcr.microsoft.com/dotnet/aspnet:8.0
WORKDIR /app
COPY publish/ .
EXPOSE 8080
CMD ["dotnet", "MyApi.dll"]
Что делает каждая команда:
FROM — базовый образ .NET Runtime;
WORKDIR — рабочая папка;
COPY — копирование опубликованного приложения;
EXPOSE — открытие порта;
CMD — запуск приложения.
После создания Dockerfile выполняется команда:
docker build -t myapi:1.0 .
где:
build — собрать образ;
-t — имя и версия образа;
. — текущая папка с Dockerfile.
Проверка:
docker images
Результат:
REPOSITORY   TAG
myapi        1.0
3. Запуск и управление контейнером
Контейнер является работающим экземпляром образа. При его запуске для ASP.NET Core используются следующие параметры:
-d (detach): запуск в фоновом режиме.
-p (ports): назначение соответствия портов хоста и контейнера (например, чтобы приложение было доступно извне по порту 80 или 443).
-v (volumes): монтирование папок хоста или создание томов для постоянного хранения данных (например, логов или БД).
--name: присвоение контейнеру понятного имени.
Контейнер создается из образа:
docker run -d \
 --name myapi \
 -p 8080:8080 \
 myapi:1.0
где:
-d — запуск в фоне;
--name — имя контейнера;
-p — проброс портов.
4. Оркестрация и взаимодействие
Если ASP.NET Core приложение является частью распределенной системы (например, взаимодействует с СУБД MS SQL Server или Redis), используется Docker Compose:
YAML-файл: В файле docker-compose.yml описывается вся конфигурация системы: сервисы (ASP.NET Core, Redis), сети (user-defined bridge для межконтейнерного взаимодействия) и тома.
Автоматизация: Docker Compose позволяет запускать и останавливать всю группу связанных контейнеров одной командой.
Healthcheck: В конфигурации можно указать точки проверки «здоровья» контейнера, что критично для работы балансировщиков нагрузки (например, Ocelot или NGINX).
Для хранения логов, файлов или базы данных используются тома (Volumes).
Пример:
docker run -d \
 -v logs:/app/logs \
 myapi:1.0
Здесь данные сохраняются даже после удаления контейнера.
5. Техническая база
Стабильность работы контейнеризированного приложения обеспечивается механизмами ядра Linux: Namespaces(изоляция сети, процессов, монтирований) и Cgroups (ограничение ресурсов процессора и памяти). Использование многослойной файловой системы OverlayFS позволяет разным образам (например, нескольким микросервисам на ASP.NET Core) использовать общие нижние слои, не дублируя их на диске.





21. Docker: контейнеризация GO-приложения.
Контейнеризация приложений на языке Go в среде Docker подчиняется общим принципам платформы, но имеет специфические особенности, связанные со сборкой исполняемых файлов под целевую операционную систему.
1. Подготовка и компиляция приложения
Для того чтобы Go-приложение могло работать внутри стандартного Docker-контейнера (который обычно базируется на ядре Linux), его необходимо правильно скомпилировать:
Целевая архитектура: Согласно источникам, Go-сервер должен быть скомпилирован специально под платформу linux-amd64.
Это позволяет получить автономный исполняемый файл, который не требует наличия установленной среды разработки Go внутри контейнера.
Пример компиляции:
GOOS=linux GOARCH=amd64 go build -o server
где:
GOOS=linux — целевая ОС Linux;
GOARCH=amd64 — архитектура x64;
server — итоговый исполняемый файл.
В отличие от ASP.NET Core, где выполняется публикация через dotnet publish и получается .dll, в Go результатом является готовый бинарный файл, который можно запускать без установленной среды Go.
После компиляции:
project/
├─ server
├─ Dockerfile
└─ ...
2. Применение Dockerfile
Dockerfile — это текстовая инструкция, на основе которой создается Docker-образ (image). Для Go-приложения в Dockerfile описываются следующие шаги:
Выбор базового образа (например, легковесного дистрибутива Linux).
Копирование скомпилированного бинарного файла приложения в файловую систему образа.
Определение команды запуска приложения при старте контейнера.
Пример для Go-приложения:
FROM ubuntu:24.04
WORKDIR /app
COPY server .
EXPOSE 8080
CMD ["./server"]
Что происходит:
берется базовый Linux-образ;
копируется скомпилированный файл server;
открывается порт;
запускается приложение.
3. Сборка и создание образа
Построение образа: Выполняется команда сборки на основе подготовленного Dockerfile. В результате создается статический шаблон (image), состоящий из неизменяемых слоев (lowerdir).
Тегирование: Для удобства управления образу можно назначить алиас (псевдоним).
Пример для Go-приложения:
FROM ubuntu:24.04

запускается приложение.
4. Запуск и управление контейнером
Контейнер является работающим экземпляром созданного образа и имеет дополнительный верхний слой для записи данных (upperdir).
Запуск: Контейнер запускается командой docker run.
Параметры: При запуске часто используются специфические параметры, такие как фоновый режим (-d), публикация портов (-p) для доступа к серверу извне и присвоение имени (--name).
После подготовки Dockerfile создается образ:
docker build -t goserver:1.0 .
Проверка образов:

Пример результата:

goserver     1.0
5. Распространение приложения
После того как образ Go-приложения создан и протестирован локально, он может быть распространен через реестры:
Публикация: С помощью команд docker login и docker push образ загружается в репозиторий, например, в Docker Hub.
Развертывание: На целевом сервере достаточно выполнить команду запуска, и Docker автоматически скачает нужный образ из репозитория и создаст на его основе работающий контейнер.

Работа Go-приложения в контейнере обеспечивается механизмами ядра Linux: Namespaces изолируют сетевые ресурсы и идентификаторы процессов (PID), а Cgroups ограничивают потребление ресурсов (процессор, память). Благодаря стандарту OCI (Open Container Initiative), такие контейнеры являются переносимыми и могут запускаться в различных средах, поддерживающих этот стандарт.



22. Docker: сети: bridge, host, none, user-defined bridge.
Docker предоставляет несколько типов сетей, которые определяют, каким образом контейнеры взаимодействуют между собой, с хостовой операционной системой и внешней сетью. Выбор сети влияет на безопасность, производительность и удобство развертывания приложений.
1. Bridge (мост по умолчанию)
Bridge — это стандартная сеть Docker, которая используется по умолчанию при запуске обычного контейнера через docker run. Docker создает виртуальный мост между контейнером и хостом, а контейнер получает собственный IP-адрес внутри этой сети. Такой режим подходит для одиночных контейнеров, но у стандартного bridge есть ограничение: контейнеры обычно не могут удобно обращаться друг к другу по именам, поэтому чаще требуется IP-адрес или явная настройка связи.
Bridge — это стандартная сеть Docker, которая используется автоматически при запуске контейнера без указания параметра --network.
Docker создает виртуальный сетевой мост между контейнером и хостом, а каждому контейнеру назначает собственный внутренний IP-адрес. Для доступа к приложению снаружи необходимо выполнять проброс портов через параметр -p.
Пример:
docker run -d -p 8080:80 nginx
В этом случае веб-сервер Nginx работает на 80-м порту контейнера, а пользователь обращается к нему через:
http://localhost:8080
Преимущества:
простая настройка;
хорошая изоляция контейнеров;
подходит для большинства одиночных приложений.
Недостаток:
контейнерам сложнее находить друг друга по именам;
чаще используется для отдельных контейнеров, а не для целых систем.
2. Host
Host — это режим, при котором контейнер не получает отдельную сетевую изоляцию, а использует сетевой стек хостовой операционной системы напрямую. То есть если приложение внутри контейнера слушает порт 3000, этот порт сразу используется на хосте. Такой режим быстрее и проще по сети, но менее безопасен, потому что контейнер почти не отделен от сети хоста. Запуск указывается через параметр --network host.
В режиме host контейнер использует сетевой стек хостовой системы напрямую.
Отдельный IP контейнеру не выдается. Приложение внутри контейнера открывает порты непосредственно на компьютере пользователя.
Пример:
docker run --network host nginx
После запуска Nginx будет доступен сразу на:
http://localhost:80
без использования -p.
Преимущества:
отсутствуют дополнительные сетевые преобразования;
выше производительность сети.
Недостатки:
практически нет сетевой изоляции;
возможны конфликты портов;
ниже уровень безопасности.
Используется, когда важна максимальная производительность или требуется прямой доступ к сетевым интерфейсам.
3. None
None — это режим полной сетевой изоляции. У контейнера отсутствует подключение к сети, кроме внутреннего loopback-интерфейса. Такой вариант применяют, когда контейнеру не нужен интернет и связь с другими сервисами, например для полностью изолированных вычислений или тестов.
Сеть контейнера полностью отключается.
Контейнер запускается без доступа к интернету, другим контейнерам и хостовой системе.
Пример:
docker run --network none ubuntu
После запуска внутри контейнера будет доступен только интерфейс:
lo (loopback)
Преимущества:
максимальная изоляция;
высокая безопасность.
Применение:
выполнение автономных вычислений;
обработка конфиденциальных данных;
тестирование программ без сетевого взаимодействия.
4. User-defined bridge (пользовательская мостовая сеть)
User-defined bridge — это пользовательская мостовая сеть, которую разработчик создает сам. Она удобнее стандартного bridge, потому что контейнеры внутри такой сети могут обращаться друг к другу по именам сервисов или контейнеров. Например, контейнер с приложением может обратиться к Redis просто по имени redis, без ручного поиска IP-адреса. В Docker Compose такие сети обычно создаются автоматически для всех сервисов из docker-compose.yml.
Это наиболее удобный тип сети для современных проектов.
Сначала создается собственная сеть:
docker network create bstu
Затем контейнеры подключаются к ней:
docker run -d --name redis --network bstu redis
docker run -d --name app --network bstu myapp
Теперь приложение может обращаться к Redis не по IP, а просто по имени контейнера:
redis:6379
Docker автоматически выполняет DNS-разрешение имен.
Преимущества:
контейнеры видят друг друга по именам;
удобна для микросервисов;
лучше изоляция между проектами;
гибкая настройка подсетей и IP-адресов.
Пример из Docker Compose
Пусть есть веб-приложение и база данных Redis.
services:
  app:
    image: myapp
  redis:
    image: redis

При запуске:
docker compose up

Docker автоматически создаст сеть типа user-defined bridge и подключит к ней оба контейнера.
Тогда приложение сможет подключиться к Redis по адресу:
redis:6379
без указания IP-адреса.

Вывод: на практике чаще всего используются bridge для простых контейнеров и user-defined bridge для реальных проектов с несколькими сервисами. Режим host применяется при необходимости прямого доступа к сети хоста, а none — когда требуется полная сетевая изоляция контейнера.




#7. Работа с томами (volumes) | Docker для начинающих
23. Docker: применение томов памяти
для понимания видос глянуть маст хев
Тома (Volumes) в Docker представляют собой специальный механизм хранения данных, который позволяет сохранять информацию независимо от жизненного цикла контейнера. Поскольку контейнеры являются временными объектами и могут создаваться, останавливаться и удаляться в любой момент, хранение важных данных непосредственно внутри контейнера считается плохой практикой. Для решения этой проблемы используются тома, которые существуют отдельно от контейнеров и могут повторно подключаться к ним при необходимости.
1. Определение и назначение
Основное назначение томов заключается в обеспечении постоянного хранения данных (Persistent Storage). Если контейнер удалить и создать заново, все данные, записанные внутрь самого контейнера, будут потеряны. Однако информация, находящаяся в томе, сохранится и может быть подключена к новому контейнеру.
Особенно важны тома для баз данных. Стандартная файловая система Docker использует механизм OverlayFS и технологию Copy-on-Write (CoW), которая хорошо подходит для приложений, но менее эффективна при интенсивной работе с большими файлами баз данных. Поэтому для PostgreSQL, MySQL, MS SQL Server и других СУБД рекомендуется использовать тома, обеспечивающие более высокую производительность и надежность хранения.
Еще одной важной особенностью является возможность совместного использования томов. Один и тот же том может быть подключен сразу к нескольким контейнерам, что позволяет организовать общий доступ к данным между сервисами без необходимости передачи файлов через сеть.
2. Типы подключения данных
Docker поддерживает два основных способа подключения внешнего хранилища к контейнеру.
Bind Mount (монтирование каталога хоста) предполагает подключение конкретной папки операционной системы к каталогу внутри контейнера. В этом случае пользователь полностью контролирует расположение файлов на хостовой машине. Такой подход часто используется во время разработки, когда необходимо мгновенно видеть изменения файлов внутри контейнера.
docker run -v C:\Projects\site:/app nginx
Например, папка проекта на компьютере может быть подключена внутрь контейнера веб-сервера, и любые изменения исходного кода будут сразу доступны приложению.
docker run -v /var/www:/app
Контейнерный каталог /app напрямую связан с папкой хоста /var/www.
Что происходит:
Хост                 Контейнер
/var/www   <---->    /app
Если ты создашь файл:
/var/www/index.html
то внутри контейнера он сразу появится как:
/app/index.html
И наоборот.
Плюсы:
удобно для разработки;
можно редактировать файлы обычным редактором на компьютере;
полный контроль над расположением данных.
Минусы:
контейнер начинает зависеть от структуры файловой системы хоста;
сложнее переносить между серверами.
Named Volume (именованный том) создается и управляется самим Docker. Пользователь указывает только имя тома, а Docker самостоятельно определяет место его хранения. 
Создание тома:
docker volume create db_data
Подключение:
docker run -v db_data:/var/lib/mysql mysql
Такой подход считается более удобным и безопасным для хранения рабочих данных приложений и баз данных.
docker run -v myvol:/app
Docker создает специальное хранилище с именем:
myvol
и сам решает, где физически хранить данные.
Схема:
Volume myvol  <---->  /app
Контейнер пишет данные в /app, а Docker сохраняет их в томе myvol.
Плюсы:
данные переживают удаление контейнера;
легко переносить между контейнерами;
удобно для БД.
Anonymous Volume (анонимный том)
docker run -v /app
или в Dockerfile: VOLUME /app
Ты не указываешь имя тома.
Docker создает том автоматически:
fd79gfzgd...
или
8d4f91e7a...
Получается:
Anonymous Volume  <----> /app
Фактически это тот же Named Volume.
Разница только в том, что имени нет.
Вместо: myvol
получаешь: fd79gfzgd98shf...
Поэтому потом сложно понять:
Какой том кому принадлежит?
Именно поэтому анонимные тома используют редко.

/ Совместное использование томов
Один том может быть подключен сразу к нескольким контейнерам.
Например:
docker run -d --name app1 -v shared_data:/data ubuntu
docker run -d --name app2 -v shared_data:/data ubuntu
Оба контейнера будут видеть одинаковые файлы в каталоге /data.
Это позволяет организовать обмен данными между сервисами без использования сети.
3. Использование томов в Docker Compose
При работе с несколькими контейнерами управление томами обычно выполняется через Docker Compose.
В файле docker-compose.yml существует отдельная секция volumes, в которой описываются именованные тома, используемые приложением. Затем в секции конкретного сервиса указывается, какой том и в какую директорию контейнера необходимо подключить.
Такой подход позволяет описывать всю инфраструктуру проекта декларативно в одном файле и автоматически создавать необходимые хранилища при запуске системы.
Кроме локальных томов, Docker Compose позволяет использовать уже существующие внешние тома (external), а также задавать дополнительные параметры хранения через драйверы томов.
В Docker Compose тома обычно описываются отдельно.
Пример:
services:
 db:
   image: mysql
   volumes:
     - db_data:/var/lib/mysql

volumes:
 db_data:
При запуске:
docker compose up
Docker автоматически создаст том db_data и подключит его к контейнеру MySQL.
4. Практическое применение томов
Наиболее распространенной областью применения томов являются системы управления базами данных. Например, при использовании MySQL или MS SQL Server файлы базы данных размещаются в томе, благодаря чему данные не теряются после обновления контейнера или его пересоздания.
Тома также активно используются в Redis для хранения снимков данных на диске и последующего восстановления состояния системы после перезапуска.
В веб-серверах, таких как NGINX, через тома часто подключаются конфигурационные файлы, SSL-сертификаты и каталоги со статическим контентом. Это позволяет изменять настройки сервера без пересборки Docker-образа.
Кроме хранения данных, Docker предоставляет средства управления томами: их можно создавать, просматривать, подключать к контейнерам, получать информацию о параметрах и удалять при необходимости.
Пример
Создание тома:
docker volume create db_data
Подключение тома к контейнеру MySQL:
docker run -d -v db_data:/var/lib/mysql mysql
В этом случае база данных будет храниться в томе db_data. Даже если контейнер удалить и создать заново, все таблицы и записи останутся сохраненными.
База данных MySQL или MS SQL
Если база хранится внутри контейнера без тома:
Удалили контейнер → потеряли всю базу данных.
Если используется том:
Удалили контейнер → база данных сохранилась.
Создали новый контейнер → данные автоматически подключились.
Redis
Redis хранит снимки памяти (dump.rdb) в подключенном томе и может восстановить данные после перезапуска контейнера.
NGINX
Можно хранить конфигурацию и сайт на хосте:
docker run -v ./nginx.conf:/etc/nginx/nginx.conf nginx
Тогда изменение файла на компьютере сразу влияет на работу контейнера.

24. Docker: инструмент Compose, основные секции конфигурационного файла
Docker Compose — это инструмент платформы Docker, предназначенный для управления многоконтейнерными приложениями. Если обычный Docker позволяет запускать отдельные контейнеры по одной команде, то Docker Compose дает возможность описать целую систему из нескольких взаимосвязанных сервисов в одном конфигурационном файле и управлять ими как единым приложением.
Основная идея Compose заключается в принципе Infrastructure as Code — вся инфраструктура проекта (контейнеры, сети, тома, переменные окружения и зависимости) описывается в текстовом YAML-файле. Благодаря этому развертывание приложения становится быстрым, воспроизводимым и не зависит от ручной настройки сервера.
Назначение и возможности Docker Compose
Docker Compose позволяет:
описывать всю архитектуру системы в одном файле docker-compose.yml;
запускать несколько контейнеров одной командой;
автоматически создавать необходимые сети и тома;
управлять жизненным циклом приложения (запуск, остановка, перезапуск, удаление);
задавать зависимости между сервисами;
централизованно хранить настройки и параметры окружения.
Например, современное веб-приложение может состоять из трех контейнеров:
NGINX (веб-сервер);
Backend на Node.js или ASP.NET Core;
База данных MySQL или PostgreSQL.
Без Docker Compose каждый контейнер пришлось бы запускать отдельно с большим количеством параметров. Compose позволяет описать всю систему в одном файле и запускать её командой:
docker compose up
После выполнения команды Docker автоматически создаст контейнеры, тома и сети, а также свяжет сервисы между собой.
Основные секции конфигурационного файла
Конфигурация Compose хранится в YAML-файле и состоит из нескольких логических секций.
1. services (Сервисы)
Это основная и наиболее важная секция. В ней описываются все контейнеры, входящие в состав приложения.
Для каждого сервиса могут задаваться различные параметры:
image — образ Docker Hub или локальный образ;
build — сборка собственного образа из Dockerfile;
ports — публикация портов наружу;
expose — открытие портов только внутри Docker-сети;
environment и env_file — переменные окружения;
volumes — подключение томов;
networks — подключение к сетям;
depends_on — зависимости между сервисами;
restart — политика автоматического перезапуска;
healthcheck — проверка работоспособности приложения.
Например, сервис базы данных может выглядеть следующим образом:
services:
  db:
    image: mysql
    restart: always
В этом случае Compose автоматически создаст контейнер MySQL и будет перезапускать его при сбоях.
2. networks (Сети)
Секция networks предназначена для описания пользовательских сетей Docker.
По умолчанию Docker Compose автоматически создает сеть типа user-defined bridge, к которой подключаются все сервисы проекта. Благодаря этому контейнеры могут обращаться друг к другу по именам сервисов.
Например:
backend → db
Здесь приложение может обращаться к базе данных по имени db, без знания её IP-адреса.
В данной секции могут задаваться:
driver — тип сетевого драйвера (обычно bridge);
driver_opts — дополнительные параметры сети;
external — использование уже существующей сети.
Практический пример.
Если в системе работают контейнеры:
backend;
redis;
mysql.
Все они могут находиться в одной сети и взаимодействовать друг с другом по именам:
redis:6379
mysql:3306
3. volumes (Тома)
Секция volumes используется для объявления именованных томов.
Тома позволяют хранить данные отдельно от контейнеров и сохранять их даже после удаления контейнера.
Для томов могут задаваться:
driver (обычно local);
external (использование заранее созданного тома).
Практический пример.
Пусть контейнер MySQL хранит базу данных в томе:
db_data
Если контейнер удалить и создать заново, данные клиентов, заказов и товаров сохранятся, поскольку они находятся в томе, а не внутри контейнера.
Именно поэтому тома практически всегда используются для:
MySQL;
PostgreSQL;
Redis;
MongoDB;
файлов пользователей.
4. secrets (Секреты)
Секция secrets предназначена для хранения конфиденциальных данных.
С её помощью можно передавать в контейнер:
пароли;
API-ключи;
токены доступа;
сертификаты.
Это более безопасный подход, чем хранение секретов непосредственно в коде приложения или в открытых переменных окружения.
Пример использования:
Пароль от базы данных передается контейнеру через секрет и не отображается в конфигурации приложения.
5. configs (Конфигурации)
Секция configs позволяет подключать внешние конфигурационные файлы к контейнерам.
Это удобно, когда необходимо хранить настройки отдельно от образа.
Практический пример.
Для контейнера NGINX можно подключить собственный файл:
nginx.conf
Тогда изменение конфигурации не потребует пересборки Docker-образа.
Пример реального проекта
Представим интернет-магазин, состоящий из трех сервисов:
NGINX — принимает HTTP-запросы пользователей;
Backend — обрабатывает бизнес-логику;
MySQL — хранит данные.
Docker Compose:
создаст три контейнера;
создаст общую сеть для их взаимодействия;
создаст том для хранения базы данных;
подключит конфигурационные файлы;
обеспечит автоматический запуск всей системы одной командой.
В результате вместо десятков команд администратор выполняет только:
docker compose up -d
и получает полностью готовое приложение.
Вывод
Docker Compose является инструментом для управления многоконтейнерными приложениями и позволяет декларативно описывать всю инфраструктуру проекта в одном YAML-файле. Основными секциями являются services, networks, volumes, secrets и configs. Использование Compose значительно упрощает развертывание, сопровождение и масштабирование современных приложений, состоящих из нескольких взаимодействующих контейнеров.





25. Файловый сервер: определение, назначение, WebDAV: создание и применение WebDAV-сервера
1. Определение и назначение файлового сервера WebDAV
Файловый сервер — это сервер, предназначенный для централизованного хранения, управления и предоставления доступа к файлам по сети. Пользователи могут загружать, скачивать, изменять и совместно использовать документы независимо от их физического расположения.
Одной из современных технологий организации файловых серверов является WebDAV (Web Distributed Authoring and Versioning) — расширение протокола HTTP/HTTPS, позволяющее работать с удаленной файловой системой практически так же, как с обычной локальной папкой.
В отличие от обычного веб-сервера, который в основном предоставляет доступ только для чтения данных, WebDAV поддерживает создание, изменение, удаление и совместное редактирование файлов через сеть.
Назначение WebDAV
Технология WebDAV используется для решения нескольких задач:
организация удаленных файловых хранилищ;
создание сетевых дисков через интернет;
совместная работа пользователей с документами;
централизованное хранение файлов организации;
управление версиями документов;
резервное хранение данных.
Фактически WebDAV можно рассматривать как альтернативу FTP, SMB и другим файловым протоколам, но использующую стандартный HTTP/HTTPS, который хорошо поддерживается сетевой инфраструктурой и средствами безопасности.
Практический пример
Предположим, в компании сотрудники работают из разных городов.
Вместо пересылки документов по электронной почте создается WebDAV-сервер:
https://files.company.by
Каждый сотрудник подключает его как сетевой диск и получает доступ к общим папкам:
Документы
Проекты
Отчеты
Шаблоны

При этом все файлы находятся на одном сервере и доступны через интернет.
2. Методы протокола WebDAV
WebDAV расширяет стандартные HTTP-команды новыми методами для работы с файлами и каталогами.
К стандартным HTTP-методам относятся:
GET — получение файла;
PUT — загрузка файла;
DELETE — удаление файла.
Однако для полноценной работы с файловой системой были добавлены специальные методы.
PROPFIND
Позволяет получать информацию об объекте:
размер файла;
дату создания;
владельца;
права доступа.
Аналог просмотра свойств файла в проводнике Windows.
PROPPATCH
Используется для изменения свойств объекта.
Например:
изменение описания файла;
изменение пользовательских атрибутов.
MKCOL
Создает новую коллекцию (каталог).
Пример:
Проекты/
 ├── 2026/
 ├── Документы/
 └── Отчеты/
Команда MKCOL позволяет создать новую папку непосредственно на сервере.
COPY и MOVE
Выполняют копирование и перемещение файлов без необходимости скачивания их на клиентскую машину.
Преимущество заключается в том, что операция выполняется непосредственно на сервере и не создает лишний сетевой трафик.
LOCK и UNLOCK
Используются для совместной работы нескольких пользователей.
Когда сотрудник начинает редактировать документ:
Отчет.docx
сервер может заблокировать файл.
Пока блокировка активна, другие пользователи не смогут одновременно изменить документ и случайно перезаписать чужие изменения.
После завершения работы выполняется команда UNLOCK.
Управление версиями
Дополнительные расширения WebDAV позволяют хранить историю изменений файлов.
Используются методы:
CHECKIN;
CHECKOUT;
VERSION-CONTROL.
Это особенно важно в системах документооборота, где необходимо отслеживать изменения документов и возвращаться к предыдущим версиям.
3. Создание WebDAV-сервера
WebDAV-сервер может быть реализован различными способами.
Наиболее популярными являются:
NGINX;
Apache;
Node.js + Express;
ASP.NET Core.
Реализация на базе NGINX
Создание WebDAV-сервера обычно включает следующие этапы:
Шаг 1. Подготовка каталога

mkdir /var/webdav

Шаг 2. Настройка пользователей
Для ограничения доступа создаются учетные записи пользователей и пароли.
Например:
Администратор
Менеджер
Разработчик
Каждый пользователь получает собственные права доступа.
Шаг 3. Настройка NGINX
В конфигурации сервера указываются:
путь к файловому хранилищу;
разрешенные WebDAV-методы;
параметры аутентификации;
использование HTTPS.
Шаг 4. Проверка и запуск

Реализация на Node.js
WebDAV можно создать программно через Express и специальные middleware.
В этом случае приложение:
принимает WebDAV-запросы;

проверяет права доступа пользователей;
выполняет операции над файлами.
Такой подход позволяет реализовать собственное облачное хранилище с индивидуальной логикой работы.

Диск Z:

Он позволяет управлять файлами через терминал.

корпоративные порталы.

а сервер выполняет соответствующие WebDAV-команды.

Шаг 3. Настройка NGINX
После создания папки для хранения файлов необходимо настроить NGINX, чтобы он обрабатывал WebDAV-запросы.
В конфигурационном файле сервера (nginx.conf или отдельный файл в sites-available) указывается:
1. Путь к файловому хранилищу
NGINX должен знать, где на диске находятся файлы пользователей.
Например:
location /webdav {
    root /var;
}
или
location /webdav {
    alias /var/webdav;
}
В этом случае все файлы будут храниться в каталоге:
/var/webdav


2. Разрешенные WebDAV-методы
По умолчанию NGINX позволяет только чтение файлов.
Для WebDAV необходимо явно разрешить операции:
dav_methods PUT DELETE MKCOL COPY MOVE;

Здесь:
PUT — загрузка файлов;
DELETE — удаление;
MKCOL — создание папок;
COPY — копирование;
MOVE — перемещение.
Без этой строки пользователь сможет только скачивать файлы.

3. Аутентификация пользователей
Чтобы любой человек из интернета не получил доступ к файлам, настраивается авторизация.
Например:
auth_basic "WebDAV";
auth_basic_user_file /etc/nginx/.htpasswd;

Файл .htpasswd содержит логины и пароли пользователей.
Пример:
ivan:$apr1$...
admin:$apr1$...

При подключении к серверу пользователь увидит окно ввода логина и пароля.

4. Использование HTTPS
Поскольку через WebDAV могут передаваться документы, логины и пароли, соединение обычно защищается SSL-сертификатом.
В конфигурации указываются:
ssl_certificate /etc/ssl/cert.pem;
ssl_certificate_key /etc/ssl/key.pem;

После этого сервер становится доступен по адресу:
https://files.company.by

а все данные передаются в зашифрованном виде.

Итоговый пример конфигурации
server {
    listen 443 ssl;

    ssl_certificate /etc/ssl/cert.pem;
    ssl_certificate_key /etc/ssl/key.pem;

    location /webdav {
        alias /var/webdav;

        dav_methods PUT DELETE MKCOL COPY MOVE;

        auth_basic "WebDAV";
        auth_basic_user_file /etc/nginx/.htpasswd;
    }
}

После такой настройки пользователь сможет:
Подключиться к серверу по HTTPS.
Ввести логин и пароль.
Создавать папки.
Загружать файлы.
Перемещать и удалять документы.
Работать с сервером как с сетевым диском.
На экзамене обычно достаточно сказать:
В конфигурации NGINX указывается каталог хранения файлов (alias или root), разрешаются WebDAV-команды через dav_methods, настраивается аутентификация пользователей (auth_basic) и защищенное соединение через SSL/TLS (ssl_certificate, ssl_certificate_key). Это позволяет превратить обычный веб-сервер в WebDAV-файловое хранилище.




26. Механизм Service Discovery: определение, назначение, применение
В современных распределенных системах и микросервисной архитектуре приложение обычно состоит не из одного сервера, а из множества отдельных сервисов. Например, интернет-магазин может включать сервис пользователей, сервис товаров, сервис заказов, сервис платежей и сервис уведомлений. Для выполнения бизнес-операций эти сервисы постоянно взаимодействуют друг с другом.
Проблема заключается в том, что в контейнерной среде адреса сервисов не являются постоянными. После перезапуска контейнера, масштабирования приложения или переноса на другой сервер IP-адрес может измениться. Если все сервисы будут обращаться друг к другу по жестко прописанным адресам, система станет сложной в сопровождении и начнет ломаться при любых изменениях инфраструктуры.
Для решения этой проблемы используется механизм Service Discovery (обнаружение сервисов) — система, которая позволяет находить нужный сервис по его логическому имени, а не по конкретному IP-адресу.
Определение и назначение
Service Discovery — это механизм автоматического обнаружения сервисов в распределенной системе, который хранит информацию о доступных экземплярах приложений и предоставляет другим компонентам их актуальные адреса.
Вместо обращения:
http://192.168.1.45:8080
приложение обращается:
http://orders-service
а система Service Discovery самостоятельно определяет, где сейчас находится данный сервис.
Основная цель механизма — избавить разработчиков и администраторов от ручного управления IP-адресами и портами в динамической инфраструктуре.
Почему Service Discovery необходим
Представим интернет-магазин.
Имеются сервисы:
User Service
Product Service
Order Service
Payment Service

Пользователь оформляет заказ.
Последовательность работы будет выглядеть следующим образом:
User Service
      ↓
Order Service
      ↓
Payment Service
      ↓
Notification Service

Если Payment Service был перезапущен и получил новый IP-адрес, остальные сервисы больше не смогут его найти.
Service Discovery автоматически обновляет информацию и сообщает всем компонентам новый адрес сервиса.
Поэтому приложение продолжает работать без изменения конфигурации.
Основные функции Service Discovery
1. Хранение реестра сервисов
Система содержит специальный реестр (Registry), где хранится информация обо всех зарегистрированных сервисах:
orders-service → 10.0.1.5:8080
users-service → 10.0.1.6:8080
payment-service → 10.0.1.7:8080
Когда сервису требуется обратиться к другому сервису, он сначала обращается к реестру.
2. Автоматическая регистрация
После запуска новый сервис автоматически сообщает о себе системе обнаружения.
Например:
payment-service
IP: 10.0.2.15
Port: 8080
Реестр обновляется без участия администратора.
3. Автоматическое удаление
Если сервис завершил работу или перестал отвечать, его запись удаляется из реестра.
Благодаря этому остальные компоненты не будут отправлять запросы на неработающий сервер.
4. Проверка работоспособности (Health Check)
Service Discovery регулярно проверяет состояние сервисов.
Например:
GET /health
Если сервис отвечает:
Status: Healthy
он остается в реестре.
Если проверки начинают завершаться ошибками, сервис исключается из списка доступных узлов.
Практический пример работы
Допустим, запущены три экземпляра сервиса заказов:
orders-service-1
orders-service-2
orders-service-3
Реестр содержит:
orders-service:
 - 10.0.1.10
 - 10.0.1.11
 - 10.0.1.12
Когда приходит запрос:
http://orders-service
Service Discovery возвращает один из доступных адресов.
Если второй экземпляр перестанет работать:
orders-service-2
он будет автоматически исключен из списка.
Оставшиеся сервисы продолжат обслуживать запросы.
Умная балансировка нагрузки
Service Discovery часто участвует в балансировке нагрузки.
При выборе сервера могут учитываться:
текущая загрузка CPU;
объем используемой памяти;
количество активных соединений;
географическое расположение клиента;
специальные теги сервисов.
Например:
EU Payment Service
US Payment Service
ASIA Payment Service
Пользователь из Европы может автоматически направляться на европейский сервер.
Канареечное тестирование (Canary Release)
Service Discovery помогает постепенно внедрять новые версии приложений.
Допустим:
orders-service v1.0
orders-service v2.0
Сначала новая версия получает только 5% запросов.
Если ошибок нет:
5% → 20% → 50% → 100%
После успешного тестирования старая версия отключается.
Это позволяет обновлять систему практически без риска для пользователей.
Поддержка нескольких версий
Во многих системах разные версии сервиса некоторое время работают одновременно.
Например:
api-v1
api-v2
Старые клиенты продолжают использовать первую версию API, а новые работают со второй.
Service Discovery помогает управлять таким переходным периодом.
Безопасность и mTLS
Современные реализации поддерживают взаимную TLS-аутентификацию (mTLS).
В обычном HTTPS клиент проверяет сервер.
В mTLS:
Сервис A проверяет Сервис B
Сервис B проверяет Сервис A

Таким образом внутренние сервисы убеждаются, что общаются именно с доверенными компонентами системы.
Это особенно важно для банковских и корпоративных систем.
Популярные реализации
Для реализации Service Discovery используются специальные платформы.
Наиболее распространены:
Consul
etcd
Netflix Eureka
Apache ZooKeeper
встроенные механизмы Kubernetes
Пример работы Consul
Consul состоит из двух основных компонентов.
Server
Хранит информацию обо всем кластере:
Список сервисов
IP-адреса
Порты
Статусы
Agent
Запускается рядом с приложением.
Он:
регистрирует сервис;
выполняет Health Check;
отправляет обновления на Server.
Интеграция с API Gateway
Очень часто Service Discovery работает вместе с API Gateway.
Например, шлюз Ocelot получает запрос:
/api/orders
Вместо статического IP он обращается к Consul:
Где находится orders-service?
Consul возвращает актуальный адрес:
10.0.2.15:8080
После этого Ocelot автоматически перенаправляет запрос.
Если сервис будет перенесен на другой сервер, настройки шлюза менять не придется.





27. Consul: реализация механизма Service Discovery, основные возможности
Consul — это платформа от компании HashiCorp, предназначенная для реализации механизма Service Discovery, мониторинга сервисов и организации безопасного взаимодействия между компонентами распределенной системы. Consul особенно широко используется в микросервисной архитектуре, контейнерных средах Docker и Kubernetes, где сервисы постоянно создаются, удаляются и меняют свои сетевые адреса.
Основная задача Consul заключается в том, чтобы избавить разработчиков и администраторов от необходимости вручную отслеживать IP-адреса и порты сервисов. Вместо этого система автоматически ведет реестр доступных сервисов и предоставляет актуальную информацию всем компонентам инфраструктуры.
1. Реализация механизма Service Discovery в Consul
Consul реализует Service Discovery через централизованный реестр сервисов.
Представим систему интернет-магазина:
User Service
Order Service
Payment Service
Notification Service
Каждый сервис может иметь несколько экземпляров:
payment-service-1
payment-service-2
payment-service-3
После запуска каждый экземпляр автоматически регистрируется в Consul.
В реестре появляется информация:
payment-service
 ├─ 10.0.1.10:8080
 ├─ 10.0.1.11:8080
 └─ 10.0.1.12:8080
Когда другому сервису требуется выполнить запрос:
http://payment-service
он обращается к Consul и получает актуальный адрес доступного экземпляра.
Таким образом сервисы взаимодействуют по логическим именам, а не по конкретным IP-адресам.
Архитектура Consul
Consul работает по клиент-серверной модели и состоит из двух основных компонентов.
Server
Серверная часть отвечает за хранение информации о состоянии кластера.
Серверы хранят:
список узлов;
список сервисов;
данные проверок здоровья;
настройки безопасности;
содержимое Key-Value хранилища.
Обычно в продакшене запускают несколько серверов для обеспечения отказоустойчивости.
Например:
Consul Server 1
Consul Server 2
Consul Server 3
Даже если один сервер выйдет из строя, остальные продолжат работу.
Agent
Agent устанавливается на каждый сервер или контейнер.
Он выполняет несколько задач:
регистрирует локальные сервисы;
выполняет Health Check;
передает информацию на серверы Consul;
принимает запросы от приложений.
Фактически Agent является посредником между сервисом и кластером Consul.
Интерфейсы доступа
Consul предоставляет два основных способа поиска сервисов.
DNS-интерфейс
Позволяет получать адрес сервиса через обычные DNS-запросы.
Например:
payment-service.service.consul
Consul возвращает IP-адрес работающего экземпляра.
Для приложений это выглядит так же, как обычный DNS.
HTTP API
Предоставляет REST-интерфейс для работы с системой.
Через API можно:
регистрировать сервисы;
получать список узлов;
читать данные KV-хранилища;
просматривать результаты проверок.
Например:
GET /v1/catalog/services
вернет список всех зарегистрированных сервисов.
2. Основные возможности Consul
Health Checking (Проверка работоспособности)
Одной из важнейших возможностей Consul является автоматический контроль состояния сервисов.
Каждый сервис периодически проверяется.
Например:
GET /health
или
TCP:3306
Если сервис отвечает корректно:
Status: Healthy
он остается в реестре.
Если сервис перестал отвечать:
Status: Critical
Consul автоматически исключает его из выдачи.
Практический пример
Имеются три экземпляра:
payment-service-1
payment-service-2
payment-service-3
Второй экземпляр аварийно завершился.
Consul обнаруживает сбой и обновляет реестр:
payment-service
 ├─ payment-service-1
 └─ payment-service-3
Запросы продолжают обслуживаться без участия администратора.
Умная балансировка нагрузки
Consul способен помогать балансировщикам нагрузки выбирать наиболее подходящий экземпляр сервиса.
При выборе могут учитываться:
загрузка CPU;
объем памяти;
регион размещения;
специальные теги.
Например:
payment-eu
payment-us
payment-asia
Пользователь из Европы может автоматически направляться к европейскому экземпляру сервиса.
Канареечное тестирование (Canary Deployment)
Consul позволяет постепенно переводить пользователей на новую версию приложения.
Допустим имеются:
api-v1
api-v2
Сначала новая версия получает:
5% трафика
Затем:
20%
50%
100%
Если ошибок не возникает, старая версия отключается.
Это снижает риск неудачных обновлений.
Key-Value Storage (KV-хранилище)
Consul содержит встроенную базу данных типа «ключ-значение».
В ней можно хранить:
параметры конфигурации;
строки подключения;
настройки приложений;
флаги функций.
Например:
database/host = mysql.local
database/port = 3306
Приложения могут получать эти значения через API без изменения своего кода.
Метаданные и теги
Каждый сервис может содержать дополнительную информацию.
Например:
payment-service
Tags:
- v2
- production
- europe
или
Meta:
region=eu
team=backend
Это позволяет выполнять поиск и маршрутизацию только к нужным экземплярам.
Безопасность и Service Mesh
Consul поддерживает концепцию Service Mesh — безопасной сети взаимодействующих сервисов.
Для этого используются специальные sidecar-прокси (обычно Envoy), которые располагаются рядом с каждым сервисом.
mTLS (Mutual TLS)
В обычном HTTPS клиент проверяет сервер.
В mTLS происходит взаимная проверка:
Order Service ↔ Payment Service
Оба сервиса подтверждают свою подлинность.
Это защищает систему от:
подмены сервисов;
несанкционированного доступа;
перехвата внутреннего трафика.
Поддержка версий сервисов
Consul позволяет одновременно использовать несколько версий приложения.
Например:
order-service-v1
order-service-v2
Старые клиенты продолжают работать с первой версией, а новые постепенно переходят на вторую.
Это называется обеспечением окна совместимости.
Web-интерфейс (UI)
Consul предоставляет удобный веб-интерфейс администратора.
Через него можно просматривать:
список серверов;
зарегистрированные сервисы;
состояние Health Check;
содержимое KV-хранилища;
состояние Service Mesh.
Например, администратор может сразу увидеть:
✓ payment-service
✓ order-service
✗ notification-service
и быстро определить источник проблемы.
Интеграция с API Gateway
Очень часто Consul работает совместно с API Gateway.
Например, шлюз Ocelot получает запрос:
/api/orders
Вместо фиксированного IP он обращается к Consul:
Где сейчас находится orders-service?
Consul возвращает актуальный адрес:
10.0.2.15:8080
После чего шлюз автоматически направляет запрос нужному сервису.
Если контейнер будет пересоздан и получит новый IP, никаких изменений в конфигурации шлюза выполнять не потребуется.




Nginx — Простым языком на понятном примере
Что такое Nginx?
1. Что такое NGINX
NGINX — это высокопроизводительный HTTP-сервер и прокси-сервер. Проще говоря, это программа, которая стоит между пользователем и веб-приложением и принимает входящие запросы. Дальше NGINX решает, что с этим запросом сделать: отдать файл, передать запрос на backend, распределить нагрузку между серверами, обработать HTTPS или выполнить роль API Gateway.
В современной веб-системе NGINX часто является первым компонентом, к которому обращается пользователь. Поэтому его называют Edge Layer — крайний уровень системы. Он находится на границе между внешним интернетом и внутренними сервисами.
2. Зачем нужен NGINX
NGINX нужен не только для “запуска сайта”. Его роль шире.
Он может работать как веб-сервер, то есть отдавать HTML, CSS, JavaScript, изображения и другие статические файлы.
Он может работать как обратный прокси, то есть принимать запросы от пользователей и перенаправлять их внутренним приложениям, например на Node.js, ASP.NET Core, Go или Python.
Он может выполнять балансировку нагрузки, распределяя запросы между несколькими экземплярами приложения.
Он может быть API Gateway, то есть единой точкой входа для микросервисной системы.
Он может обрабатывать HTTPS, принимая защищенные соединения и снимая эту нагрузку с приложения.
Также через дополнительные модули NGINX может использоваться как WebDAV-сервер для работы с файлами через сеть.
3. Как работает NGINX в общей схеме
Типичная схема такая:
Клиент → NGINX → Приложение → База данных
Пользователь не обращается напрямую к приложению. Он обращается к NGINX. А уже NGINX перенаправляет запрос дальше.
Например, пользователь открывает сайт:
site.com
NGINX принимает запрос и может сделать следующее:
если запрошен index.html — отдать файл с диска;
если запрошен /api/users — передать запрос backend-приложению;
если запрос пришел по HTTPS — расшифровать его;
если backend запущен в нескольких копиях — выбрать один из серверов.
4. NGINX как веб-сервер
Как веб-сервер NGINX умеет отдавать статические файлы.
Статические файлы — это файлы, которые не изменяются на сервере перед отправкой пользователю:
HTML;
CSS;
JavaScript;
изображения;
шрифты;
видео;
React/Vue/Angular bundle.
Например, после сборки React-приложения появляется папка dist. В ней лежат index.html, JS-файлы, CSS-файлы и картинки. NGINX может напрямую отдавать эту папку пользователю.
Это выгодно, потому что сервер приложения не тратит ресурсы на простую отправку файлов. Он занимается только бизнес-логикой, а NGINX быстро отдает статику.
5. NGINX как Reverse Proxy
Обратный прокси — это одна из главных ролей NGINX.
Reverse Proxy означает, что NGINX принимает запрос от клиента и передает его внутреннему серверу.
Например:
Клиент → NGINX → localhost:3000
Пользователь видит только домен:
example.com
Но внутри запрос может уходить на приложение:
localhost:3000
или
backend:5000
Это позволяет скрыть внутренние адреса серверов. Клиент не знает, где реально работает приложение. Он знает только NGINX.
Пример простой логики:
запрос пришел на example.com;
NGINX принял его;
NGINX отправил его на localhost:3000;
приложение вернуло ответ;
NGINX передал ответ пользователю.
6. Проксирование HTTP-запросов
Проксирование — это процесс передачи запроса от NGINX к другому серверу.
В конфигурации это обычно делается через proxy_pass.
Например:
server {
listen 80;
server_name example.com;
location / {
proxy_pass http://localhost:3000;
}
}
Это значит: все запросы к example.com нужно передавать приложению, которое работает на localhost:3000.
Можно разделять запросы:
статику NGINX отдает сам;
API передает backend-серверу.
Например:
location / {
root /var/www/site;
}
location /api/ {
proxy_pass http://localhost:3000;
}
В такой схеме запросы к сайту идут к файлам, а запросы к /api/ уходят в приложение.
7. Заголовки при проксировании
Когда NGINX передает запрос backend-приложению, важно сохранить информацию о клиенте.
Для этого используются специальные заголовки:
Host — исходный домен;
X-Real-IP — реальный IP клиента;
X-Forwarded-For — цепочка IP-адресов, через которые прошел запрос.
Без этих заголовков приложение может видеть только IP самого NGINX, а не настоящего пользователя.
8. NGINX как балансировщик нагрузки
Если приложение запущено в нескольких экземплярах, NGINX может распределять запросы между ними.
Например:
       Клиент
            ↓
       NGINX
      ↙   ↓   ↘
App1 App2 App3
В конфигурации это делается через upstream.
Например:
upstream backend {
server localhost:3000;
server localhost:3001;
server localhost:3002;
}
server {
listen 80;
location /api/ {
proxy_pass http://backend;
}
}
Теперь NGINX будет отправлять запросы не на один сервер, а распределять их между несколькими.
Это нужно для повышения производительности и отказоустойчивости. Если один сервер перегружен или недоступен, можно продолжать работу через другие.
9. NGINX как API Gateway
NGINX может выполнять роль API Gateway.
API Gateway — это единая точка входа в систему, особенно если система состоит из нескольких микросервисов.
Например:
/api/users → User Service
/api/orders → Order Service
/api/payments → Payment Service
Для клиента все выглядит как один API, но внутри NGINX распределяет запросы по разным сервисам.
Пример:
location /api/users/ {
proxy_pass http://user-service:5001/;
}
location /api/orders/ {
proxy_pass http://order-service:5002/;
}
location /api/payments/ {
proxy_pass http://payment-service:5003/;
}
Здесь NGINX смотрит на путь запроса и решает, в какой сервис его отправить.
10. NGINX и SPA-приложения
SPA — это Single Page Application, то есть одностраничное приложение. Например, React, Vue или Angular.
После сборки SPA превращается в набор статических файлов:
index.html;
main.js;
style.css;
assets.
NGINX идеально подходит для размещения SPA.
Он отдает index.html, JavaScript и CSS, а запросы к API отправляет backend-приложению.
Схема:
Пользователь → NGINX → SPA-файлы
и отдельно:
SPA → /api → NGINX → Backend
Важный момент: в SPA маршруты часто обрабатываются на стороне браузера. Например:
/profile
/settings
/course/5
Физически таких папок на сервере может не быть. Поэтому NGINX обычно настраивают так, чтобы при неизвестном маршруте он все равно отдавал index.html, а уже React/Vue разбирался с маршрутом.
11. NGINX и HTTPS
NGINX часто отвечает за HTTPS.
HTTPS — это защищенный HTTP, где данные передаются через TLS-шифрование.
Для HTTPS нужны:
SSL/TLS-сертификат;
закрытый ключ;
настройка виртуального сервера на порт 443.

Схема:
Клиент
↓ HTTPS
NGINX
↓ HTTP
Backend
То есть пользователь подключается к NGINX по HTTPS, а NGINX уже может передавать запрос внутреннему приложению по обычному HTTP.
Это называется SSL/TLS-терминация.
Преимущество в том, что сертификаты и шифрование настраиваются в одном месте. Не нужно отдельно настраивать HTTPS в каждом микросервисе.
12. Конфигурация NGINX
Главный файл конфигурации обычно находится здесь:
/etc/nginx/nginx.conf

Также часто используются отдельные файлы в:
/etc/nginx/sites-available
/etc/nginx/sites-enabled
В конфигурации описывается:
какой порт слушать;
какой домен обслуживать;
где лежат статические файлы;
куда проксировать запросы;
какие сертификаты использовать;
какие логи писать;
какие методы разрешены.

После изменения конфигурации обычно выполняют:
nginx -t
Эта команда проверяет синтаксис.

Если ошибок нет:
nginx -s reload
Эта команда применяет новую конфигурацию без полной остановки сервера.
13. Серверные процессы NGINX
NGINX работает через процессную архитектуру.
Есть Master Process.
Он отвечает за:
чтение конфигурации;
запуск worker-процессов;
перезагрузку сервера;
управление рабочими процессами.

Есть Worker Process.
Они непосредственно обрабатывают запросы пользователей.
Обычно количество worker-процессов делают равным количеству ядер процессора.

Например:
4 ядра → 4 worker-процесса
Такая архитектура позволяет NGINX эффективно обслуживать большое количество одновременных соединений.
14. Модули NGINX
NGINX расширяется через модули.
Модули добавляют поддержку разных возможностей:
HTTP-сервер;
проксирование;
SSL/TLS;
gzip-сжатие;
кэширование;
FastCGI;
uWSGI;
SCGI;
WebDAV.
FastCGI, SCGI и uWSGI используются для взаимодействия с приложениями и фреймворками, особенно в связке с Python, PHP и другими backend-технологиями.
WebDAV-модуль позволяет использовать NGINX как файловый сервер, где пользователи могут не только скачивать файлы, но и загружать, удалять, перемещать и создавать папки.
15. Журналы NGINX
У NGINX есть два основных типа логов.
Access Log — журнал доступа.
Туда записывается информация о запросах:
IP клиента;
дата и время;
HTTP-метод;
URL;
код ответа;
размер ответа;
user-agent.

По access.log можно понять:
кто заходил на сайт;
какие страницы открывали;
какие API вызываются чаще всего;
есть ли подозрительная активность.

Error Log — журнал ошибок.
Туда попадают:
ошибки конфигурации;
ошибки подключения к backend;
проблемы с SSL;
ошибки доступа к файлам;
внутренние ошибки NGINX.
Если сайт не работает, error.log — одно из первых мест, куда смотрит администратор.
16. NGINX и Docker
NGINX часто разворачивают в Docker.
Можно взять официальный образ nginx и передать в контейнер свою конфигурацию и файлы сайта.
Обычно используются:
-p — проброс портов;
-v — подключение конфигурации или файлов с хоста;
-d — запуск в фоне.
Например, можно пробросить порт 80 контейнера наружу, а папку со статикой подключить как volume.
В Docker NGINX часто работает как reverse proxy перед другими контейнерами.
Например:
nginx → frontend
nginx → backend
nginx → api-service
В Docker Compose NGINX обычно описывается как отдельный сервис, который зависит от backend-сервисов и находится с ними в одной Docker-сети.
17. NGINX и WebDAV
WebDAV — это расширение HTTP, которое позволяет работать с файлами на удаленном сервере почти как с обычной сетевой папкой.
Обычный HTTP в основном позволяет получать данные:
GET — получить файл;
POST — отправить данные.

А WebDAV добавляет файловые операции:
PUT — загрузить файл;
DELETE — удалить файл;
MKCOL — создать папку;
COPY — скопировать файл;
MOVE — переместить файл.
То есть WebDAV превращает веб-сервер в файловое хранилище.
Пользователь может подключить WebDAV как сетевой диск и работать с файлами через проводник Windows, файловый менеджер Linux или специальные клиенты.
18. Настройка WebDAV в NGINX
Для WebDAV нужно указать каталог, где будут храниться файлы.
Например:
location /webdav {
alias /var/webdav;
}
Затем нужно разрешить WebDAV-методы:
dav_methods PUT DELETE MKCOL COPY MOVE;
Без этого пользователь сможет только читать файлы, но не сможет загружать, удалять или создавать папки.
Также обязательно нужна аутентификация, чтобы любой человек из интернета не получил доступ к файлам.
Например:
auth_basic "WebDAV";
auth_basic_user_file /etc/nginx/.htpasswd;
Файл .htpasswd хранит логины и пароли пользователей.
Так как через WebDAV могут передаваться документы, логины и пароли, его обычно настраивают только через HTTPS.
19. Пример применения WebDAV
Допустим, в организации есть отделы:
бухгалтерия;
отдел кадров;
разработка.
На сервере создаются папки:
/Accounting
/HR
/Development
Сотрудники подключают WebDAV как сетевой диск и работают с документами.
Преимущества:
файлы хранятся централизованно;
доступ контролируется правами;
можно делать резервное копирование;
пользователи работают с файлами через привычный интерфейс;
все операции выполняются на сервере.
20. Итоговое понимание NGINX
NGINX — это не просто “сервер для сайта”. Это универсальный инфраструктурный компонент, который стоит на входе в веб-систему и управляет трафиком.
Он может:
отдавать статические файлы;
размещать SPA-приложения;
проксировать запросы к backend;
скрывать внутренние серверы;
балансировать нагрузку;
работать как API Gateway;
обрабатывать HTTPS;
вести логи;
разворачиваться в Docker;
работать как WebDAV-файловый сервер.
Если сказать максимально просто, NGINX — это диспетчер входящих запросов. Пользователь обращается не напрямую к приложению, а к NGINX. А NGINX уже решает, что делать: отдать файл, передать запрос на API, выбрать нужный сервер, обеспечить HTTPS или дать доступ к файлам через WebDAV.




Что такое Docker?
Docker Для Начинающих за 1 Час | Docker с Нуля
1. Что такое Docker и зачем он появился
До Docker разработчики постоянно сталкивались с проблемой:
«У меня на компьютере работает, а на сервере нет».
Причина заключалась в различиях окружения:
разные версии библиотек;
разные версии ОС;
разные настройки сервера;
отсутствующие зависимости.
Docker решает эту проблему через контейнеризацию.
Контейнеризация — это способ упаковать приложение вместе со всеми его зависимостями в изолированную среду, которая будет одинаково работать на любом компьютере, где установлен Docker.
Получается принцип:
Build once — Run anywhere.
Собрали приложение один раз и запускаем где угодно.
2. Контейнеризация и виртуализация
Очень любят спрашивать различие.
Виртуальная машина:
Железо
↓
Гипервизор
↓
Гостевая ОС
↓
Приложение
Каждая VM содержит собственную операционную систему.

Контейнер:
Железо
↓
Хостовая ОС
↓
Docker Engine
↓
Контейнеры
Контейнеры используют ядро хостовой ОС.

Поэтому:
запускаются быстрее;
занимают меньше места;
потребляют меньше памяти;
проще масштабируются.
Именно поэтому Docker стал стандартом для микросервисов.
3. Архитектура Docker
Docker состоит из нескольких компонентов.
Docker Engine
Основной движок платформы.
Отвечает за:
создание контейнеров;
запуск контейнеров;
управление сетями;
управление томами;
управление образами.
Docker Daemon
Фоновый сервис.
Именно он реально создает контейнеры.
Команды пользователя идут сначала к Docker CLI, потом к Docker Daemon.
Docker CLI
Интерфейс командной строки.
Например:
docker run
docker build
docker ps
Через него пользователь управляет Docker.
Docker Registry
Хранилище образов.
Самое известное:
Docker Hub
Оттуда можно скачать готовые образы:
nginx;
mysql;
redis;
postgres;
node;
golang.
4. Образ (Image)
Образ — это шаблон будущего контейнера.
Можно представить его как ISO-файл операционной системы.
Образ содержит:
приложение;
библиотеки;
конфигурацию;
инструкции запуска.
Сам по себе образ не выполняется.
Это просто заготовка.
Например:
nginx image
node image
postgres image
5. Контейнер (Container)
Контейнер — это запущенный экземпляр образа.
Аналогия:
Класс → Объект
Image → Container
Один образ может породить множество контейнеров.
Например:
nginx image
↓
Container 1
Container 2
Container 3
Все созданы из одного образа.
6. Стандарт OCI
OCI (Open Container Initiative) — открытый стандарт контейнеров.
Он определяет:
формат образов;
правила запуска;
структуру контейнеров.
Благодаря OCI образ может работать не только в Docker, но и в других контейнерных системах.
7. Жизненный цикл контейнера
Контейнер может находиться в нескольких состояниях:
Created
Создан, но не запущен.
Running
Работает.
Stopped
Остановлен.
Removed
Удален.
Основные действия:
запуск;
остановка;
перезапуск;
удаление.
8. Запуск контейнера
При запуске Docker:
Находит образ.
Если образа нет — скачивает его.
Создает контейнер.
Настраивает сеть.
Запускает процесс внутри контейнера.
Каждый контейнер обычно имеет один основной процесс.
Например:
nginx → nginx process
postgres → postgres process
9. Порты контейнера
Контейнер живет в собственной сети.
Поэтому его порт 80 не виден снаружи.
Нужно выполнять проброс портов.
Например:
Хост:80 → Контейнер:80
Тогда пользователь сможет обращаться к приложению извне.
Без проброса контейнер доступен только внутри Docker.
10. Dockerfile
Dockerfile — это инструкция по созданию собственного образа.
Он описывает:
базовый образ;
копирование файлов;
установку зависимостей;
команду запуска.
Docker читает Dockerfile сверху вниз и формирует новый образ.
Именно Dockerfile используется для контейнеризации приложений.
11. Создание образа
Процесс выглядит так:
Исходный код
↓
Dockerfile
↓
docker build
↓
Image
↓
docker run
↓
Container
То есть сначала создается образ, потом запускается контейнер.
12. Контейнеризация ASP.NET Core
Для ASP.NET Core обычно используется двухэтапная сборка.
Первый этап:
компиляция проекта;
публикация приложения.
Второй этап:
копирование готовых файлов;
запуск через ASP.NET Runtime.
Преимущество:
SDK остается только на этапе сборки.
В итоговом образе находится только готовое приложение.
Образ становится меньше.
13. Контейнеризация Go-приложения
У Go есть особенность.
После компиляции получается самостоятельный бинарный файл.
Поэтому процесс проще.
Сначала приложение компилируется под Linux:
linux-amd64
Затем исполняемый файл помещается внутрь контейнера.
В результате контейнер может содержать практически только:
Linux;
бинарный файл программы.
Поэтому Go-контейнеры обычно очень маленькие.
14. Распространение приложений
Docker сильно упростил доставку приложений.
Раньше:
Исходный код
↓
Настройка сервера
↓
Установка библиотек
↓
Запуск
Теперь:
docker pull
↓
docker run
Все зависимости уже находятся внутри образа.
15. Сети Docker
Каждый контейнер получает собственное сетевое пространство.
Docker поддерживает несколько типов сетей.
Bridge
Самая популярная сеть.
Контейнеры находятся в общей виртуальной сети.
Могут обращаться друг к другу по имени.
Используется по умолчанию.
Host
Контейнер использует сеть хостовой машины напрямую.
Нет сетевой изоляции.
Плюс:
максимальная производительность.
Минус:
меньше безопасности.
None
Сеть полностью отключена.
Контейнер не имеет сетевого доступа.
Используется редко.
User Defined Bridge
Пользовательская bridge-сеть.
Самый популярный вариант для реальных проектов.
Позволяет контейнерам обращаться друг к другу по DNS-именам.
Например:
nginx
backend
database
Могут взаимодействовать по именам сервисов.
16. Тома (Volumes)
Контейнеры считаются временными.
Если контейнер удалить, его файловая система исчезнет.
Поэтому нужны тома.
Volume — это постоянное хранилище данных.
Схема:
Контейнер
↓
Volume
↓
Диск сервера
Даже если контейнер удалить и создать заново, данные сохранятся.
17. Где используются тома
Самый частый пример — база данных.
PostgreSQL:
Контейнер
↓
Volume
↓
Файлы БД
Если контейнер удалить, база не потеряется.
Также тома используют для:
логов;
конфигураций;
пользовательских файлов;
резервных копий.
18. Docker Compose
Когда контейнеров становится много, запускать их вручную неудобно.
Например:
NGINX

Redis
Тогда используют Docker Compose.
Compose позволяет описать всю систему в одном YAML-файле.
После этого вся инфраструктура запускается одной командой.

24. Итоговое понимание Docker
Docker — это платформа контейнеризации, которая позволяет упаковать приложение вместе со всеми зависимостями в изолированный контейнер. Основные сущности Docker — Image (образ), Container (контейнер), Network (сеть), Volume (том) и Docker Compose (оркестрация нескольких контейнеров). 1. Что такое API Gateway
API Gateway (шлюз API) — это единая точка входа в распределенную систему или микросервисную архитектуру.
Вместо того чтобы клиент обращался напрямую к каждому сервису, он работает только с Gateway.
Без Gateway:
Клиент
├── User Service
├── Order Service
├── Payment Service
├── Notification Service
Клиент должен знать:
адрес каждого сервиса;
порт каждого сервиса;
способы авторизации;
особенности каждого API.
Это неудобно.

С Gateway:
Клиент
↓
API Gateway
↙ ↓ ↘
Users Orders Payments
Клиент знает только один адрес.
Всю остальную работу выполняет Gateway
2. Почему появился API Gateway
Когда система становится микросервисной, появляются проблемы.
Например, есть:
User Service
Order Service
Payment Service
Review Service
Notification Service
Frontend должен обращаться сразу ко всем сервисам.
Получаются десятки запросов.
Кроме того:
сервисы могут менять адреса;
сервисы могут масштабироваться;
сервисы могут временно падать;
нужно централизованно проверять безопасность.
API Gateway решает эти проблемы.
3. Главная идея API Gateway
Главная идея очень простая:
"Клиент не должен знать внутреннее устройство системы."
Клиент работает только с Gateway.
Внутри можно менять:
сервисы;
порты;
количество серверов;
инфраструктуру.
Для клиента ничего не изменится.
4. Основная функция №1 — маршрутизация
Маршрутизация (Routing) — это главная задача Gateway.
Gateway смотрит на URL запроса и определяет, какому сервису его отправить.
Например:
/api/users
↓
User Service
/api/orders
↓
Order Service
/api/payments
↓
Payment Service
Клиенту не нужно знать адреса этих сервисов.
Gateway знает их сам.
5. Основная функция №2 — сокрытие инфраструктуры
Допустим сервис пользователей работает так:
http://10.0.0.15:5001
Сервис заказов:
http://10.0.0.16:5002
Пользователь никогда не должен видеть эти адреса.
Он видит только:
https://api.company.com
Вся внутренняя инфраструктура оказывается скрыта.
Это повышает безопасность.
6. Основная функция №3 — аутентификация
Представим систему из 20 микросервисов.
Без Gateway каждый сервис должен самостоятельно проверять:
JWT;
OAuth;
OpenID Connect;
права доступа.
Получается дублирование кода.
С Gateway:
Клиент
↓ JWT
Gateway
↓
Микросервисы
Gateway проверяет токен один раз.
Если токен валиден — запрос пропускается дальше.
Если нет — сразу возвращается ошибка.
Это называется централизованная аутентификация.
7. Основная функция №4 — авторизация
После проверки личности Gateway может проверять права пользователя.
Например:
Администратор → доступ разрешен
Обычный пользователь → доступ запрещен
То есть Gateway может решать, какие сервисы доступны конкретному пользователю.
8. Основная функция №5 — балансировка нагрузки
Очень часто сервис существует не в одном экземпляре.
Например:
User Service 1
User Service 2
User Service 3
Gateway может распределять запросы между ними.
Схема:
Gateway
↙ ↓ ↘
User1 User2 User3
Пользователь даже не знает, какой сервер обработал запрос.
Этим занимается Gateway.
9. Алгоритмы балансировки
Gateway может использовать различные алгоритмы.
Round Robin
Запросы идут по кругу.
1 → Server1
2 → Server2
3 → Server3
4 → Server1
Least Connection
Выбирается сервер с наименьшим количеством активных соединений.
Sticky Session
Пользователь закрепляется за конкретным сервером.
Это важно для stateful-приложений.
IP Hash
Один и тот же IP всегда попадает на один сервер.

для понимая
Схема такая:
Клиент открывает:
https://api.site.com/api/users
Дальше:
api.site.com → DNS → IP балансировщика → один из API Gateway → нужный микросервис
То есть клиент не знает про конкретные API Gateway. Он знает только домен api.site.com.
по сути, если я правильно понял, именно днс определяет куда что пойдет, в плане, он делает запрос - в ответе у него несколько ip балансировщиков, и он по тому же принципу выбирает, кому отдать запрос. самый распространенный - облачный балансировщик
вопрос уже кто как заморочится и у кого сколько будет узлов балансировки и гетвей
10. Основная функция №6 — агрегация запросов
Очень важная функция.
Представим страницу профиля пользователя.
Для ее открытия нужны данные из:
User Service
Order Service
Payment Service
Без Gateway:
Frontend делает 3 запроса.
С Gateway:
Frontend делает 1 запрос.
Gateway сам обращается ко всем сервисам и собирает ответ.
Это называется Aggregation.
11. Основная функция №7 — Rate Limiting
Иногда необходимо ограничить количество запросов.
Например:
не более 100 запросов в минуту.
Если пользователь превышает лимит:
Gateway возвращает ошибку.
Это защищает систему от:
перегрузок;
ботов;
DDoS;
злоупотребления API.
12. Основная функция №8 — логирование
Все запросы проходят через Gateway.
Поэтому очень удобно вести логи именно здесь.
Можно записывать:
пользователя;
IP;
URL;
время ответа;
ошибки.
Получается единое место мониторинга системы.
13. Основная функция №9 — мониторинг
Gateway может показывать:
количество запросов;
нагрузку;
ошибки;
состояние сервисов.
Это облегчает администрирование.
14. Circuit Breaker
Представим:
Payment Service упал.
Без Circuit Breaker:
Gateway продолжает слать запросы.
Сервис получает еще большую нагрузку.
Ситуация ухудшается.
С Circuit Breaker:
Gateway замечает сбой.
Временно перестает отправлять запросы.
Ждет восстановления сервиса.
Потом постепенно возвращает трафик.
Это называется "Прерыватель цепи".
15. Service Discovery
Еще одна важная тема.
В Docker и Kubernetes адреса сервисов могут постоянно меняться.
Например:
User Service сегодня:
10.0.0.15
Через минуту:
10.0.0.27
Gateway не может хранить адреса вручную.
Поэтому используется Service Discovery.
Специальная система (например Consul) сообщает Gateway актуальные адреса сервисов.
Gateway всегда знает, куда отправлять запрос.
16. Backend For Frontend (BFF)
Иногда разные клиенты требуют разные данные.
Есть:
веб-приложение;
мобильное приложение;
Smart TV.
Для каждого можно создать собственный Gateway.
Это называется Backend For Frontend (BFF).
Каждый клиент получает API, адаптированное именно под него.
17. Реализации API Gateway
Инфраструктурный уровень:
NGINX
HAProxy
Apache HTTP Server
Они работают как reverse proxy и могут выполнять роль Gateway.
Уровень приложения (.NET):
Ocelot
YARP
Это полноценные API Gateway внутри ASP.NET Core.
Облачные решения:
Azure API Management
AWS API Gateway
Они предоставляются как готовый сервис.
18. Отличие API Gateway от Reverse Proxy
Reverse Proxy умеет:
принимать запросы;
перенаправлять запросы.
API Gateway умеет всё то же самое плюс:
аутентификация;
авторизация;
балансировка;
агрегирование;
логирование;
мониторинг;
rate limiting;
circuit breaker.
То есть Gateway можно считать "умным" Reverse Proxy.
19. Отличие API Gateway от Load Balancer
Load Balancer решает только одну задачу:
распределение нагрузки.
Gateway делает гораздо больше:
маршрутизацию;
безопасность;
балансировку;
агрегацию;
мониторинг.
Поэтому Load Balancer — это лишь одна из функций Gateway.
20. Итоговое понимание API Gateway
API Gateway — это единая точка входа в микросервисную систему, которая принимает все запросы клиентов и берет на себя инфраструктурные задачи: маршрутизацию запросов, аутентификацию, авторизацию, балансировку нагрузки, агрегацию данных, ограничение запросов, логирование, мониторинг, Service Discovery и отказоустойчивость через Circuit Breaker. Клиент взаимодействует только с Gateway, а вся сложность внутренней архитектуры скрывается внутри системы.
